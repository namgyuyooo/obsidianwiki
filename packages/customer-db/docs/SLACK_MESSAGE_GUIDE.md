# Slack 메시지 작성 가이드

`#tf_cross_team_sales` 메시지가 고객 DB에 더 잘 정리되도록 돕는 작성 제안입니다.  
Slack 메시지는 GLM이 구조화한 뒤 DB에 들어가며, 애매하거나 기존 DB와 충돌하는 값은 사용자 정합성 확인 요청으로 넘어갈 수 있습니다.

## 작성할 때 조금만 도와주면 좋은 점

1. 가능하면 한 메시지에는 한 고객사 또는 한 활동만 적어 주세요.
2. `회사명`, `발생일시`, `관심 솔루션`처럼 필드명을 붙여 주면 자동 정리에 도움이 됩니다.
3. 확인된 사실과 추정은 살짝 구분해 주세요.
4. 이메일, 회사명, 날짜, 관심 솔루션이 있으면 DB 매칭 정확도가 좋아집니다.
5. 개인정보/연락처는 업무상 필요한 범위만 공유해 주세요.

## 최근 `#tf_cross_team_sales` 기준 개선 포인트

최근 메시지들은 대체로 `고객명`, `방문 일자`, `참석자`, `방문 목적`, `결과`, `미팅 및 논의 내용`, `Next Plan` 구조가 잘 잡혀 있어 GLM이 고객 활동으로 해석하기 좋습니다. 아래 항목을 조금만 더 명확히 적어 주면 DB 입력과 정합성 확인 부담을 줄일 수 있습니다.

- `참석자` 안의 고객 담당자는 가능하면 `이름 / 부서 / 직급 / 이메일 / 휴대폰`으로 나눠 주세요.
- `고객명: PSKH, PSK`처럼 여러 회사를 한 필드에 넣어야 할 때는 주 고객사와 관련사를 나눠 주면 좋습니다.
- `26.07.08`처럼 짧은 날짜도 이해할 수 있지만, 가능하면 `2026-07-08` 형식을 부탁드립니다.
- `EHM 1copy 발주 검토`, `Hubble 검사 시스템 추진`, `RISA 관심`처럼 관심 솔루션을 별도 필드로 한 번 더 적어 주면 좋습니다.
- `TODO`, `Next Plan`에는 담당 주체를 붙여 주세요. 예: `RTM: 자료 발송`, `고객: 데이터 공유`.
- 이미지나 파일을 올릴 때는 `회사명`, `활동내용`, `첨부파일 의미`를 한 줄만 덧붙여도 DB 정리에 큰 도움이 됩니다.
- `보임`, `추측`, `가능성`, `미확정` 같은 내용은 `확인상태: 확인 필요`로 적어 주면 이후 확인 요청으로 자연스럽게 넘길 수 있습니다.

## 확인상태는 이렇게 적어 주면 좋습니다

- `확인됨`: 고객이 직접 말했거나 명함/문의폼/문서에 명확히 있는 값
- `확인 필요`: 들은 내용이지만 기존 DB와 다르거나, 회사/부서/직급이 불확실한 값
- `추정`: 이메일 도메인, 검색, 문맥으로 추정한 값

`확인 필요` 또는 `추정`으로 적어 주면 DB에 바로 확정하지 않고 정합성 확인 요청으로 넘길 수 있습니다.

## 관심 솔루션 표기

가능하면 아래 이름 중 하나로 적어 주세요.

- `Hubble`
- `EHM`
- `RISA`
- `M.AX Agent`
- `TS Agent`
- `기타`

여러 개면 쉼표로 구분해 주시면 됩니다.

```text
관심 솔루션: Hubble, EHM
```

## 조금만 보완하면 좋은 예

```text
케이씨텍 쪽에서 연락옴. 비전검사 관심 있는 듯. 나중에 봐야 함.
```

이 메시지는 회사명은 잡을 수 있지만 이메일, 담당자, 활동일시, 관심 솔루션, 다음 액션을 확정하기 어렵습니다. 아래 필드 중 아는 것만 몇 개 더 붙여 주면 충분합니다.

```text
ABC랑 PSK 같이 미팅함. Hubble도 하고 EHM도 얘기함. 자료 보내야 할 듯.
```

이 메시지는 어느 회사의 관심 솔루션인지, 누가 무엇을 해야 하는지, 새 리드인지 기존 활동인지 분리하기 어렵습니다. 회사별로 블록을 나눠 주면 GLM이 더 안정적으로 정리할 수 있습니다.

```text
Reflow Ball Top View
```

이미지나 파일과 함께 이 한 줄만 있으면 DB에는 고객명, 솔루션, 활동내용이 남기 어렵습니다. `회사명`과 `첨부파일 의미`만 덧붙여도 충분히 좋아집니다.

## 이렇게 적으면 더 잘 정리됩니다

| 상황 | 가능하면 이렇게 | 보완이 필요한 형태 |
| --- | --- | --- |
| 고객명 | `회사명: ABC 기업` | `ABC쪽`, `고객사`, `PSKH, PSK`처럼 한 필드에 복수 회사 |
| 날짜 | `발생일시: 2026-07-08 14:00` | `26.07.08`, `오늘`, `금일`만 작성 |
| 담당자 | `김ABC / 품질팀 / 팀장 / kim@example.com` | 참석자 이름만 나열 |
| 관심 솔루션 | `관심 솔루션: Hubble, EHM` | 본문 중간에만 “허블도 얘기함” |
| 결과 | `결과: EHM 1copy 발주 검토` | `좋은 반응`, `가능성 있음` |
| 다음 액션 | `RTM: 자료 발송`, `고객: 데이터 공유` | `추후 확인`, `나중에 대응` |
| 불확실한 정보 | `확인상태: 확인 필요`, `확인 필요 사유: 일정 미확정` | 추정 내용을 확정처럼 작성 |
| 첨부파일 | `첨부파일: Reflow Ball 이미지. PoC 검토용 샘플` | 파일만 업로드 |

## DB 정합성을 위해 있으면 좋은 값

- 신규 리드: `회사명`, `이메일`, `관심 솔루션`, `문의내용`
- 기존 고객 활동: `회사명` 또는 `이메일`, `활동내용`, `다음 액션`
- 회사 정보 변경: `회사명`, 바뀐 필드, `근거`, `확인상태`
- 방문/미팅 결과: `발생일시`, `회사명`, `고객 담당자`, `관심 솔루션`, `결과`, `다음 액션`, `확인상태`

## GLM이 DB로 변환하는 기준

- `[신규 리드]`는 `lead`
- `[고객 활동]`은 `activity`
- `[회사 정보 업데이트]`는 `company_update`
- 잡담, 단순 알림, 내부 확인만 있는 메시지는 `ignore`

필드가 불완전하거나 기존 DB와 다르면 `consistency_reviews`에 쌓이고, 사용자가 승인/수정/거절한 뒤 본 DB에 반영됩니다.
