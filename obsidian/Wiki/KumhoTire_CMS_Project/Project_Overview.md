## 기존 정리 메모
금호타이어 CMS 프로젝트는 공개 Slack 기준으로 타이어 검사 시각화와 측정 로직을 실제 배포 환경에서 검증하고 수정하는 운영 전환형 프로젝트다. `금호타이어 첫제품 X-ray`와 고객은 같지만, 이 채널은 X-ray 납품 흐름보다 CMS 시각화/배포/해상도 이슈 대응에 더 집중되어 있어 별도 workstream으로 보는 편이 자연스럽다.

채널 메시지에는 2025년 12월 배포 내용과 2026년 4월 배포 후 테스트 결과가 모두 남아 있다. 특히 `PL1H`, `CENTER_LINE`, `HBW1/2 시각화 개선`, `폰트 크기 수정` 같은 화면/로직 변경 사항과 `result.zip`, `raw/overlay` 이미지 결과물이 함께 확인된다.

## 한 줄 요약

- 금호타이어 CMS 프로젝트는 배포된 검사 시각화/측정 로직을 현장 데이터 기준으로 다시 검증하고 보정하는 프로젝트다.

## 현재까지 확인된 실제 정보

- 2025년 12월 배포 내역 공유 메시지에서 다음 변경 사항이 확인된다.
  - `PL1H 추가`
  - `CENTER_LINE 추가`
  - `HBW1, 2 시각화 개선`
  - `폰트 크기 수정`
- 2026년 4월 15일 배포 후 테스트 결과에서는 FTP 기존 데이터를 기준으로 테스트했고, `in/out 폴더`와 함께 결과 파일이 공유됐다.
- 같은 테스트에서 `0012026071460101Z1`이 실패 케이스로 따로 지목된다.
- 2026년 4월 중순 메시지에서는 연구소 데이터가 아니라 공장 데이터처럼 해상도가 높은 조건에서는 결과가 거의 맞지 않는다는 피드백이 나온다.
- 후속 메시지에서는 학습 당시 `300 x 300` 고정 crop 크기를 사용했다는 설명과, 해상도 증가 시 모델 입력을 동적으로 조정했음에도 결과가 충분히 안정적이지 않다는 맥락이 확인된다.
- 2026년 1월 회의록에서는 공장과 연구소의 니즈가 다르며, 공장에서는 작업자가 중요하게 보는 8가지 항목 중심으로 먼저 맞추는 방향이 정리된다.
- 같은 회의록에서 공장 데이터와 연구소 데이터의 품질 차이가 커서, 실제 적용을 생각하면 공장 이미지 기반으로 다시 학습하는 시나리오가 더 현실적이라는 의견이 나온다.
- 2026년 1월 23일 논의에서는 Capply 검출 정확도가 기대보다 낮아, 우선 `belt1~2 사이 거리`를 이용한 수식 기반 UA 계산으로 대응하고 추후 contour 로직 고도화 시 vision 로직을 반영하는 방향이 제시된다.
- 배포 일정은 금호 측 내부 보고 시점을 고려해 1월 3주를 사실상 데드라인으로 인식하고 원격 배포까지 검토한 흔적이 있다.

## 핵심 결정

- 시각화 화면 개선은 단순 cosmetic 수정이 아니라 측정 기준선과 텍스트 가독성까지 포함하는 방향으로 진행됐다.
- FTP 기준 테스트 데이터를 기준으로 배포 검증을 반복하는 운영 방식이 사용됐다.
- 해상도 이슈는 단순 데이터 문제로 넘기지 않고 모델/입력 구조 수정 여부를 같이 검토하는 방향으로 논의됐다.
- 공장 적용을 우선할 경우 연구소 이미지보다 공장 이미지 기반 개선이 더 중요하다는 인식이 형성됐다.
- 일부 항목은 AI 검출보다 규칙/수식 기반 계산을 우선 적용하고, 이후 비전 로직 고도화로 넘어가는 단계적 접근이 사용됐다.

## 현재 상태

- 배포와 테스트는 이미 여러 차례 진행된 상태다.
- 하지만 공장 데이터 기준 해상도 상승 환경에서는 성능 불안정성이 남아 있다.
- 따라서 이 프로젝트는 “배포 완료”보다는 “배포 후 현장 적합성 보정 단계”로 보는 편이 적절하다.

## 위키에 남겨야 할 포인트

- 같은 금호타이어 계열이라도 `첫제품 X-ray`와 `CMS`는 증적 축이 다르므로 문서를 분리해두는 편이 좋다.
- 이 프로젝트의 핵심은 모델 자체보다 `현장 데이터 해상도 변화`와 `시각화/측정 로직 보정`이다.
- 향후 다른 비전 프로젝트에서도 학습 환경과 운영 환경의 해상도 차이가 핵심 리스크로 재사용될 수 있다.
- 공장과 연구소가 중요하게 보는 항목이 다르다는 점도 현장 전환형 비전 프로젝트의 공통 리스크로 남길 가치가 높다.
