# RTM Government RnD Complete Template

## 목적

- 이 문서는 알티엠 정부과제 초안을 가장 빠르게 만들기 위한 `완성본 템플릿`이다.
- 단순 문단 골격이 아니라 `바로 붙여넣을 초안`, `과제별 교체 슬롯`, `문서 유형별 분기 규칙`을 함께 제공한다.
- 새 과제가 들어오면 이 문서를 복사한 뒤 `{슬롯}`만 교체하고, 실제 수치와 기관명만 근거 문서에 맞게 보정하는 방식으로 사용한다.

## 사용 순서

1. `{사업명}`, `{과제명}`, `{수요기업}`, `{대상 공정/업무}`, `{핵심 문제}`, `{실증 환경}`을 먼저 채운다.
2. 문서 유형을 고른다.
3. 아래 완성본 초안에서 해당 유형 문장으로 교체한다.
4. KPI와 레퍼런스 수치는 원문 근거에 맞게 보정한다.
5. 마지막으로 `과장형 표현 제거`, `AS-IS/TO-BE`, `실증 경로`, `사업화 구조`가 빠지지 않았는지 확인한다.

## 문서 유형 분기

### 운영형 연구개발계획서

- 대표 상황: 산업현장, PSK, 설비 운영/유지보수, Fab 현장
- 강조 요소:
  - 데이터 분절
  - 실시간 운영 의사결정
  - 예지보전/이상탐지/조치 권고
  - 대체 실증 경로

### 상용화 사업계획서형

- 대표 상황: 서울반도체, 공정 최적화, 도입형 사업계획서
- 강조 요소:
  - `AS-IS / TO-BE`
  - 블랙박스 공정
  - 센서/설비/데이터 인프라
  - 도입 후 비용/수율/운영 효과

### 공급형 기술사업화형

- 대표 상황: 한양대, 기술사업화, On-Prem 공급형
- 강조 요소:
  - On-Premise 패키지
  - RAG 업데이트
  - 지식 그래프/검색/추론
  - 라인 단위 공급과 갱신 매출

### 서비스형 오케스트레이션형

- 대표 상황: XaaS, AI바우처, 멀티에이전트 서비스
- 강조 요소:
  - 중앙 오케스트레이터
  - 전문 에이전트 분업
  - 실행형 액션
  - 운영 KPI와 플랫폼 확산

## 슬롯 정의

- `{사업명}`
- `{과제명}`
- `{수요기업}`
- `{대상 공정/업무}`
- `{대상 설비/라인}`
- `{핵심 문제}`
- `{운영상 문제}`
- `{데이터 종류}`
- `{실증 환경}`
- `{주관기관 역할}`
- `{참여기관 역할}`
- `{정량 KPI}`
- `{1차 확산 대상}`
- `{2차 확산 대상}`
- `{사업화 구조}`

## 목차별 완성본 초안

### 1. 연구개발 필요성

```markdown
{사업명} 대상 {대상 공정/업무}는(은) {산업 변화/시장 압력}이 확대되는 상황에서도 여전히 {핵심 문제}로 인해 운영 효율과 품질 안정화에 구조적 한계를 보이고 있음. 특히 {수요기업} 현장은 {데이터 종류}가 분절되어 있고, {운영상 문제}가 반복되어 현장 엔지니어가 이상 상황을 통합적으로 판단하고 즉시 대응하기 어려운 구조임.

현재 운영 방식은 단순한 편의성의 문제가 아니라 {사업/기술/경쟁력 의미}와 직결되는 문제이며, 기존의 수기 대응 또는 단일 시스템 중심 운영만으로는 대응 속도와 정확도를 동시에 확보하기 어려움. 따라서 본 과제는 {대상 공정/업무}의 현장 데이터를 기반으로 {해결 방향}을 구현하여, 실증 가능한 수준의 운영형 AI 지원 체계를 구축하는 것을 목표로 함.
```

### 2. 최종 목표

```markdown
본 과제의 최종 목표는 {수요기업}의 {대상 공정/업무}에서 발생하는 {핵심 문제}를 해결하기 위해, {데이터 종류}를 통합한 {플랫폼/AI/에이전트/시스템}을 개발하고 {실증 환경}에서 검증하는 것임.
```

### 3. 세부 목표

```markdown
1. {데이터 통합 및 관리 목표}
2. {이상 탐지/상황 인지 목표}
3. {원인 분석/예측 목표}
4. {조치 추천/제어/실행 목표}
5. {실증 및 고도화 목표}
```

### 4. 개발 내용 및 수행방법

```markdown
본 과제는 {데이터 종류}를 통합 관리하는 기반 체계를 먼저 구축하고, 이를 바탕으로 {전문 에이전트 또는 핵심 모듈}이 역할별 판단을 수행하도록 설계함. 이후 각 모듈의 결과를 {오케스트레이션/통합 운영 계층}에서 연계하여, 단순 정보 제공이 아니라 실제 운영 의사결정과 조치 지원으로 연결되는 구조를 구현하고자 함.

또한 개발 과정은 데이터 확보, 모델 개발, 운영 적용, 현장 피드백 반영 및 고도화 순으로 추진하며, {실증 환경}에서 실제 적용 가능성과 성능을 단계적으로 검증할 계획임.
```

### 5. 수행기관 역량

```markdown
알티엠은 제조현장에 특화된 시계열 AI와 비전 AI를 동시에 보유한 제조 AI 전문기업으로, `APOLLO`, `EHM`, `HUBBLE`을 중심으로 이상탐지, 품질검사, 가상계측, 예지보전, 자율운영형 의사결정 지원까지 포괄하는 기술 포트폴리오를 확보하고 있음. 또한 `PSK`, `서울반도체`, `LG에너지솔루션`, `제우스`, `픽셀` 등 실제 제조 고객사 대상 실증 및 공급 경험을 기반으로 데이터 분석, 모델 개발, 현장 적용, 유지보수까지 전주기 수행 경험을 보유하고 있어, 본 과제에서도 기술 구현과 현장 실증을 안정적으로 수행할 수 있음.
```

### 6. 참여기관 및 수요기업 역할

```markdown
알티엠은 본 과제에서 {주관기관 역할}을 담당하며, {참여기관 역할}은(는) 보완 기술과 실증 연계를 수행함. 수요기업 {수요기업}은(는) {대상 설비/라인}의 현장 문제를 제시하고, {데이터 종류}와 {실증 환경}을 제공하며, 개발 결과의 적용 가능성과 도입 효과를 검증함.
```

### 7. 실증 계획

```markdown
실증은 {실증 환경}을 기준으로 추진하며, 대상은 {대상 설비/라인}으로 설정함. 검증 과정에서는 {정량 KPI}를 중심으로 성능을 평가하고, 현장 접근이 제한되는 경우에는 테스트베드 또는 히스토리컬 로그 기반 대체 검증을 병행하여 실증 가능성을 확보할 계획임.
```

### 8. 사업화 및 확산

```markdown
본 과제 결과물은 우선 {1차 확산 대상}을 대상으로 적용하고, 이후 {2차 확산 대상}으로 단계적으로 확산 가능한 {사업화 구조}로 고도화할 계획임. 알티엠은 제조현장 맞춤형 On-Premise 패키지와 유지보수 수행 경험을 바탕으로, 단발성 실증이 아니라 라인 단위 공급과 연간 갱신/유지보수 구조로 연결되는 반복 확산 모델을 확보하고자 함.
```

### 9. 기대효과

```markdown
본 과제를 통해 {수요기업}은(는) {운영상 KPI 효과}를 기대할 수 있으며, 자료 탐색 시간 단축, 이상 대응 리드타임 감소, 숙련도 의존 완화, 품질 및 운영 의사결정 고도화 효과를 확보할 수 있음. 또한 알티엠은 현장 실증을 통해 제품 완성도와 확산 가능성을 높일 수 있으며, 과제 결과물은 후속 고객 확산과 유사 공정 적용의 기반 레퍼런스로 축적될 수 있음.
```

## 문서 유형별 교체 문장

### 운영형 연구개발계획서 교체용

- 필요성 보강:
  - `공정지식과 실시간 설비 데이터의 분절`
  - `사후 대응 중심의 유지보수 체계`
  - `비계획 정지를 계획 정비로 전환`
- 개발내용 보강:
  - `이상탐지 -> 원인 탐색 -> 예측 -> 조치 -> 학습`
  - `각 모듈은 상호 독립 기능이 아니라 연속 체계를 형성`

### 상용화 사업계획서형 교체용

- 필요성 보강:
  - `AS-IS / TO-BE 비교표`
  - `실시간 가시성 결여로 인한 블랙박스 공정`
  - `300개 이상의 다변량 센서`
- 기대효과 보강:
  - `수율 개선`
  - `원가 절감`
  - `운영 표준화`

### 공급형 기술사업화형 교체용

- 개발내용 보강:
  - `On-Premise 환경 기반 데이터 관리 및 RAG 업데이트 관리`
  - `Graph-aware RAG 기반 지식 그래프 연계 추론`
  - `이동식 저장매체 기반 업데이트 프로세스`
- 사업화 보강:
  - `라인 단위 공급 + 연간 갱신/유지보수`
  - `Recurring Revenue 구조`

### 서비스형 오케스트레이션형 교체용

- 개발내용 보강:
  - `중앙 오케스트레이터 기반 통합 운용`
  - `전문 에이전트 분업 구조`
  - `실행까지 연결되는 Agentic AI`
- 기대효과 보강:
  - `응답시간 단축`
  - `운영 자동화`
  - `플랫폼 충성도 및 후속 확장`

## 최종 점검 체크리스트

- 기관명만 바뀌고 내용은 비슷하게 복제된 문장이 없는가
- `문제 -> 구조적 이유 -> 해결 방향 -> 검증 계획` 흐름이 살아 있는가
- `실증 환경`, `정량 KPI`, `사업화 구조`가 모두 들어갔는가
- 알티엠 어필이 `필요성`에 과도하게 튀지 않는가
- `실증 결과`, `기대효과`, `계획 수치`가 혼용되지 않았는가
