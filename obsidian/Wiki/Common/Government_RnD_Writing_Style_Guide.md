## 공통 문체 특징

- 문제를 먼저 제시하고, 그 다음 현재 한계, 제안 기술, 실증 계획, 확산 계획 순으로 밀어갑니다.
- 기술 설명은 `무엇을 개발하는가`보다 `어떤 운영 문제를 어떻게 줄이는가`에 더 강하게 연결합니다.
- 수치, 기간, 기관 역할, 실증 환경을 한 문단 안에 붙여 써서 실행 가능성을 강조합니다.
- 주장형 표현보다 계획형 표현을 선호합니다.
  - `구축함`, `개발함`, `확보함`, `실증할 계획임`, `확산하고자 함`
- 현재 수준과 목표 수준을 `AS-IS / TO-BE`, `현재 / 향후`, `기존 / 고도화 후` 구조로 대비합니다.
- 최종본으로 갈수록 `양식 준수`보다 `현장 적용 단위`, `실증 시나리오`, `사업화 구조`, `발표용 메시지`가 더 선명해지는 경향이 있습니다.
- 최종본은 기능 목록만 나열하지 않고 `이상탐지 -> 원인 탐색 -> 예측 -> 조치 -> 학습`처럼 연쇄 흐름으로 기능을 묶어 설명하는 경향이 강합니다.
- 로컬에서 직접 확인한 본문은 `구축함`보다 `~하도록 설계함`, `~역할을 수행하도록 함`, `~동시에 만족하는 구조를 구현함` 같은 설계 의도형 문장 비중이 높습니다.

## 문단 전개 패턴

### 1. 배경/필요성

- 산업 변화 -> 현장 문제 -> 기존 방식 한계 -> 과제 필요성 순으로 전개
- 자주 쓰는 연결:
  - `지속 성장과 함께`
  - `단순한 운영 편의의 문제가 아니라`
  - `핵심 경쟁력의 축을 선점하는 문제임`
  - `현재와 같은 이유로 어려움을 겪고 있음`

### 2. 문제 정의

- 현장 pain point를 모호하게 쓰지 않고 `데이터 분절`, `숙련자 의존`, `실시간 관측 불가`, `비계획 정지`, `대응 지연`처럼 운영 단위로 쪼갭니다.
- 한 문제당 한 결과를 붙입니다.
  - `대응 효율 저하`
  - `MTTR 증가`
  - `유지보수 비용 누적`
  - `품질 편차 확대`

### 3. 목표/개발내용

- `최종 목표` 다음에 `세부 목표`를 번호형으로 분해하는 패턴이 강합니다.
- 세부 목표는 대체로 아래 순서를 따릅니다.
  - 데이터 통합
  - 이상 탐지 또는 상황 인지
  - 원인 분석 또는 예측
  - 조치 추천 또는 제어
  - 피드백 기반 고도화
- 로컬 최종본에서는 각 목표를 모듈명이 아니라 `현장 의사결정에서 맡는 역할`로 먼저 설명합니다.
  - `상황인지`
  - `위험도와 원인 후보 제시`
  - `잔여수명 및 정비 우선순위 산출`
  - `근거 문서 기반 조치 대안 제시`
  - `실행 결과 학습`

### 4. 실증/상용화

- 연차별 목표를 `1차년도 기반 구축 -> 2차년도 성능 고도화 -> 3차년도 현장 실증/확산` 구조로 씁니다.
- 실증은 항상 대상 고객, 대상 설비, 대상 라인, 검증 지표를 함께 적습니다.
- 기술사업화/상생형 최종본에서는 `라인 단위 공급`, `연간 갱신`, `기존 고객 확산`, `해외 실증` 같은 공급 구조가 같이 붙습니다.
- 산업현장 계열 최종본은 `현장 접근 가능 시 파일럿`, `제약 시 테스트베드/히스토리컬 로그 대체 검증`처럼 대체 실증 경로를 함께 적어 실행 가능성을 높입니다.

### 5. 확산/사업화

- 단순 매출 목표보다 `기존 고객 확산`, `계열사 확산`, `해외 PoC`, `표준 패키지화` 같은 확산 경로를 먼저 적는 문서가 강했습니다.

### 6. 보안/운영

- On-Premise, 폐쇄망, 권한관리, 로그, 승인 절차, 롤백 구조를 별도 강조하면 정부 R&D 문맥과 잘 맞습니다.
- PSK/기술사업화 계열 최종본에서는 `Fab 테스트베드`, `도메인 데이터 10만 건 이상`, `현장 베타테스트` 같은 운영 검증 표현이 반복됩니다.

## 최종본 기반 추가 패턴

### 운영형 연구개발계획서

- `운영`과 `유지보수`를 별도 주제로 나누지 않고 하나의 의사결정 체계로 묶는 문장이 강합니다.
- `실시간 운영`, `유지보수`, `Fab 테스트베드`, `현장 베타테스트`를 한 문단에 같이 배치하는 패턴이 반복됩니다.
- `각 모듈은 상호 독립 기능이 아니라 연속 체계를 형성`하는 식으로 모듈 간 연결성을 먼저 강조하고, 개별 기능은 그 다음에 풉니다.
- `단순 저장이 아니라`, `단순 이상 감지에 그치지 않고`, `단순 병렬 표시가 아니라` 같은 대비형 부정 문장이 자주 사용됩니다.
- 기능 설명 끝에는 `출발점 역할`, `지원하도록 함`, `동시에 만족하는 구조`처럼 운영상 의미를 한 번 더 덧붙입니다.

### 사업수행계획서형

- XaaS 계열은 `사업수행계획서`라는 제목 자체를 전면에 두고, 본문보다 제출형 구조와 발표자료 정합성을 강하게 의식합니다.
- 초안 버전 표기가 많이 남아 있어도 최종본으로 가면 발표본과 핵심 메시지가 일치하도록 다듬어집니다.
- AI바우처/서비스형은 `중앙 오케스트레이터`, `전문 에이전트`, `실행형 액션`, `Model-Portable` 같은 시스템 구조 키워드를 요약문 단계부터 전면에 둡니다.

### 기술사업화/상생형

- `기술 구현`과 `사업화 구조`를 별도 장으로 떼기보다, 계획서 안에서 공급 구조까지 바로 연결하는 경향이 있습니다.
- `라인 단위 On-Prem 공급`, `연간 갱신/유지보수`, `5개년 누적 매출` 같은 표현은 사업화/확산 장뿐 아니라 목표/기대효과 장에도 침투합니다.

### 상용화 사업계획서형

- 서울반도체 계열은 연구개발계획서보다 `사업계획서` 문맥이 강해서, `도입기업 현황 -> AS-IS 문제 -> TO-BE 목표시스템 -> 확산/지원계획` 순이 더 또렷합니다.
- `AS-IS / TO-BE`, `현 수준 / 구현 목표`, `도입 전 / 도입 후` 비교표를 강하게 요구하며, 비교 서술이 곧 설득의 핵심이 됩니다.

## 로컬 직접 추출 기반 세부 관찰

### 산업현장 최종 pdf

- 요약문에서 이미 `최종 목표 -> 세부 목표 -> 도입하고자 하는 AI 에이전트 -> 핵심 달성목표 -> 최종 결과물` 순서가 완성되어 있습니다.
- 문제 정의는 기술 부족보다 `공정지식과 실시간 설비 데이터의 분절`, `사후 대응 중심 유지보수`, `숙련자 의존`, `학습 환류 부재`처럼 운영 병목을 나열합니다.
- 에이전트 설명은 이름보다 역할 중심입니다.
  - 데이터 지식 융합 기반 플랫폼
  - 이상탐지 AI 에이전트
  - 예지보전 AI 에이전트
  - PE 협업 AI 에이전트
  - 오케스트레이션 및 VM 기반 제어
- 연차별 서술은 기관명을 앞에 두고, 그 뒤에 산출물과 검증 지표를 짧게 붙이는 패턴이 강합니다.

### PSK 로컬 HWPX 작업본

- 계획서 초안 단계에서도 평가 항목을 의식한 문장 삽입이 많아 `정부지원 필요성`, `도전성`, `사업화 계획`, `경제성`을 한 본문 안에 선반영합니다.
- `4M`, `버티컬 온톨로지`, `온디바이스 VLM`, `뉴로-심볼릭 AI`, `Value Chain` 같은 키워드를 통해 기술 차별성을 한 문단 안에서 압축합니다.
- `세정 공정 성공 모델 선점 -> 8대 공정 확산`처럼 테스트베드와 확산 전략을 한 세트로 서술하는 방식이 반복됩니다.

### 설반 로컬 HWPX

- `블랙박스 공정`, `실시간 가시성 결여`, `300개 이상의 센서`, `장비별·라인별 데이터 파편화`처럼 현장 문제를 매우 구체적인 공정 언어로 씁니다.
- 비교 서술이 강합니다.
  - `AS-IS / TO-BE`
  - `현 수준 / 구현 목표`
  - `도입 전 / 도입 후`
- 설비와 데이터 자산을 설명할 때 `SECS/GEM`, `288개 센서`, `10PB 규모 데이터`, `온프레미스 서버 및 AI 가속기`처럼 인프라 근거를 먼저 깔고 그 위에 AI를 얹습니다.

### 기술사업화 로컬 PDF

- 요약문에서 곧바로 `End Product`, `주요 개발 내용`, `주요 결과물`, `사업화 전략`으로 넘어가며, 결과물의 소프트웨어 단위를 명시합니다.
- `RAG 업데이트`, `에이전틱 검색`, `Graph-aware RAG`, `이동식 저장매체 기반 업데이트`처럼 폐쇄망 운영을 전제로 한 기능 설명이 두드러집니다.
- 사업화는 모호한 시장 확대보다 `라인 수`, `갱신 매출`, `Recurring Revenue`로 숫자화하는 패턴이 강합니다.

### XaaS/AI바우처 로컬 HWPX

- 요약문부터 `중앙 오케스트레이터`, `전문 에이전트`, `실행형 액션`, `오토스케일링`, `국산 AI반도체 전환`이 한 세트로 등장합니다.
- `단일 챗봇 또는 단일 예측모델 수준이 아니라` 같은 대비형 문장을 써서 시스템 구조의 차이를 분명히 합니다.
- 수요기업 문제를 `초동 수요 폭증`, `이벤트 기반 스파이크`, `고변동 구조`, `반복 CS`처럼 운영 이벤트 언어로 쓰는 점이 재사용 가치가 높습니다.

## 문장 연결 습관

- 첫 문장은 대부분 현장 문제를 규정하고, 둘째 문장에서 그 문제가 왜 반복되는지 구조적 이유를 붙입니다.
- 셋째 문장부터는 `이를 위해`, `아울러`, `단순 ~가 아니라`, `이를 통해` 같은 연결어로 해결 체계를 전개합니다.
- 마지막 문장은 기능 설명으로 끝내지 않고 `실증 가능`, `운영 가능`, `상용화 가능` 같은 귀결로 닫는 것이 안정적입니다.
- 상용화형 사업계획서에서는 마지막 문장을 `비용 절감`, `응답시간 단축`, `재구매율`, `반복 매출` 같은 운영성과로 닫는 비중이 더 높습니다.

## 서술 톤 규칙

- 단정형 홍보 문구보다 `근거 기반`, `운영형`, `실증형` 톤을 유지합니다.
- `세계 최초`, `압도적`, `혁신적` 같은 과도한 형용사는 줄이고, 대신 아래를 씁니다.
  - `현장 적용 가능`
  - `검증 가능`
  - `확산 가능`
  - `재현 가능`
  - `운영 가능한 수준`

## 작성 문체 규칙

- 문장은 `배경 설명 -> 문제 명시 -> 해결 방향 -> 기대 결과` 순으로 짧게 끊어 씁니다.
- 한 문단에는 가능하면 `문제 1개`, `주장 1개`, `근거 1개`만 넣어 읽는 사람이 빠르게 판단할 수 있게 합니다.
- 문체는 홍보형보다 `계획형`, `실행형`, `검증형`으로 맞춥니다.
  - `보유하고 있음`
  - `수행한 경험이 있음`
  - `적용 가능함`
  - `실증할 계획임`
  - `확산하고자 함`
- 추상명사만 나열하지 말고 현장 단위를 같이 붙입니다.
  - `제조 데이터 통합`
  - `설비 이상탐지`
  - `품질 편차 저감`
  - `양산 라인 실증`
- 정량 수치는 단독으로 던지지 말고 `무슨 지표인지`, `어떤 과제에서 나온 값인지`를 같이 적습니다.
- 기관 소개 문단은 `회사 개요 -> 기술 축 -> 실적 -> 본 과제 기여` 순으로 정리합니다.
- 수요기업 문단은 `현장 문제 -> 데이터 제공 -> 실증 환경 -> 도입 효과 검증` 순으로 정리합니다.
- 같은 내용이더라도 목차에 따라 문장 끝을 조정합니다.
  - 필요성: `~이 요구됨`, `~이 필요함`
  - 목표: `~을 개발하고자 함`, `~을 확보하고자 함`
  - 수행방법: `~을 통해 구현함`, `~을 기반으로 고도화함`
  - 사업화: `~으로 확산 가능함`, `~으로 전환할 계획임`

## 목차별 작성 전략

### 1. 연구개발 필요성

- 톤: 산업 변화와 현장 문제를 연결하는 `문제 정의형`
- 핵심 전략:
  - 시장/산업 변화로 시작하고, 곧바로 현장 운영 문제로 내려옵니다.
  - `왜 지금 필요한가`를 기술 유행이 아니라 운영 손실, 인력 의존, 품질 변동, 정지 리스크로 설명합니다.
  - 알티엠 소개는 최소화하고 수요기업 pain point를 전면에 둡니다.
- 쓰면 좋은 표현:
  - `현장 적용 가능한 수준의`
  - `숙련자 의존 구조를 완화할 수 있는`
  - `실시간 대응 한계를 개선하기 위한`
  - `실시간 가시성 결여로 인한 블랙박스 공정`
  - `장비별·라인별 데이터 파편화`

### 2. 개발 목표

- 톤: `도달 상태 명시형`
- 핵심 전략:
  - 최종 목표는 한 문장으로 짧게 적고, 세부 목표는 데이터-탐지-예측-조치-실증 순으로 분해합니다.
  - 목표 문장에는 `대상`, `해결 문제`, `개발 결과물` 세 요소를 반드시 넣습니다.
  - 수치는 성능목표, 시간단축, 비용절감처럼 검증 가능한 항목으로 적습니다.

### 3. 개발 내용 및 방법

- 톤: `구현 경로 설명형`
- 핵심 전략:
  - 모듈 설명보다 `어떤 데이터를 받아 어떤 판단을 수행하고 어떤 조치를 지원하는지` 흐름으로 씁니다.
  - 단계별 개발 방법은 데이터 확보, 모델 개발, 운영 적용, 피드백 고도화 순으로 씁니다.
  - 알티엠 기술 설명은 `APOLLO/EHM/HUBBLE` 제품명보다 과제 적합 기능을 앞에 배치하고, 제품명은 뒤에서 보강합니다.
  - 서비스형 과제는 `전문 에이전트 -> 중앙 오케스트레이터 -> 실행/운영 액션` 흐름으로 쓰면 설득력이 높습니다.

### 4. 수행기관 역량

- 톤: `근거 기반 신뢰도형`
- 핵심 전략:
  - 회사 일반 소개보다 `본 과제를 왜 알티엠이 수행할 수 있는가`에 집중합니다.
  - `기술 포트폴리오`, `고객 실적`, `정량 성과`, `현장 적용 경험` 순으로 압축합니다.
  - 고객 결과보고서 기반 성과와 회사소개서 기반 수치를 섞을 때는 출처 성격을 구분합니다.
- 쓰면 좋은 구조:
  - `제조 AI 전문성`
  - `유사 과제 수행 경험`
  - `현장 실증 및 확산 경험`
  - `본 과제 직접 기여 항목`

### 5. 참여기관 역할

- 톤: `책임 분담 명시형`
- 핵심 전략:
  - 기관별로 `보유 역량`, `담당 산출물`, `실증 기여`를 각각 적습니다.
  - `지원`, `협력` 같은 모호한 표현만 쓰지 말고 실제 담당 모듈을 적습니다.
  - 알티엠은 `데이터 분석/AI 모델/운영 패키지`, 수요기업은 `현장 데이터/검증 환경/도입 평가`로 분리하는 편이 좋습니다.

### 6. 실증 계획

- 톤: `검증 시나리오형`
- 핵심 전략:
  - 대상 설비, 대상 라인, 제공 데이터, 검증 지표를 한 세트로 씁니다.
  - 실증은 기술 설명보다 운영환경 설명이 더 중요합니다.
  - `어디에서`, `무엇으로`, `무엇을 검증하는지`가 빠지지 않게 씁니다.

### 7. 사업화 및 확산 전략

- 톤: `확산 경로 제시형`
- 핵심 전략:
  - 매출 목표를 바로 쓰기보다 고객 확산 시나리오를 먼저 적습니다.
  - `기존 고객 확산 -> 유사 공정 확산 -> 표준 패키지화` 구조가 잘 맞습니다.
  - 알티엠은 유지보수, 커스텀 프로젝트, 공동개발 경험을 후속 사업화 근거로 붙입니다.
  - 기술사업화 계열은 `라인 수`, `갱신 매출`, `추가 Add-on`처럼 반복 매출 구조를 함께 적는 편이 강합니다.
  - 서비스형 과제는 `운영 효율화 -> 플랫폼 충성도 -> 후속 확장`의 순서가 자연스럽습니다.

### 8. 기대효과

- 톤: `성과 전환형`
- 핵심 전략:
  - 기술 성능을 운영 KPI로 번역합니다.
  - 정확도 향상만 적지 말고 `가동률`, `분석시간`, `불량비용`, `인력부하`, `의사결정 속도`로 연결합니다.
  - 수요기업 효과와 공급기관 효과를 섞지 말고 분리하는 편이 읽기 좋습니다.

## 재활용 시 주의

- 기관명만 바꾸고 문단을 복제하지 말고, 각 기관의 역할과 확보 데이터, 실증 환경, 기대효과만 다시 맞춥니다.
- 참여기관 문장은 `보유 역량 + 담당 모듈 + 실증 기여` 3요소가 모두 들어가야 합니다.
- 수요기업 문장은 `현장 문제 + 제공 데이터 + 검증 환경 + 도입 효과` 4요소가 들어가야 합니다.
