# Cross Project Patterns

최근 2년 공개 Slack 증적을 기준으로 보면, 여러 고객 프로젝트에서 반복되는 구조가 뚜렷하다. 이 문서는 프로젝트별 로그를 넘어 공통으로 재사용할 수 있는 운영 패턴을 모아두는 페이지다.

## 1. POC 성공과 상업 전환은 별개다

- 서울바이오시스처럼 POC 반응이 긍정적이어도 실제 도입 단계에서는 가격, 의사결정 주체, 경쟁 제품 비교가 병목이 된다.
- 따라서 `POC 결과`와 `상업 전환 조건`은 별도 페이지나 섹션으로 분리해 관리하는 편이 좋다.

## 2. 연구소 데이터와 현장 데이터는 다르다

- 금호타이어 CMS처럼 연구소 이미지와 공장 이미지의 해상도, 품질, 중요 항목이 다르면 기존 모델이 그대로 동작하지 않는다.
- 향후 비전 프로젝트에서는 `학습 데이터 기준`과 `실사용 데이터 기준`을 처음부터 구분해 기록해야 한다.

## 3. 클라우드 가능 여부와 데이터 반출 허용은 다르다

- LG엔솔처럼 AWS 사용 자체는 가능해도 고객 이미지 반출은 금지되는 경우가 있다.
- 인프라 논의에서는 `클라우드 사용 가능`, `고객 데이터 업로드 가능`, `내부 서버 대체 가능`을 따로 적는 것이 중요하다.

## 4. AI 검출이 어려우면 규칙 기반 임시 대응이 필요하다

- 금호타이어 CMS에서 보듯 일부 항목은 AI 검출 정확도가 부족할 때 수식 기반 계산으로 먼저 안정화하고, 이후 비전 로직을 고도화하는 접근이 유효하다.
- 이 패턴은 현장 일정이 촉박한 프로젝트에서 특히 자주 등장한다.

## 5. 보조 채널도 독립 프로젝트만큼 중요할 수 있다

- 픽셀의 `DMT`처럼 독립 프로젝트는 아니어도 실제 XML/NAS/CSV 운영 구조를 설명하는 보조 채널이 존재한다.
- 프로젝트 폴더 안에 하위 workstream 문서를 두면 이런 보조 채널을 자연스럽게 흡수할 수 있다.

## 6. 초기 탐색 과제는 빨리 독립 프로젝트로 올리지 않는다

- PSK 온도예측task처럼 상견례와 요구사항 청취만 있는 경우에는 별도 프로젝트보다 상위 고객 계정의 하위 workstream으로 두는 편이 안정적이다.
- 파일, 수치, 결정사항이 붙기 전까지는 승격을 보류하는 기준이 유효하다.
