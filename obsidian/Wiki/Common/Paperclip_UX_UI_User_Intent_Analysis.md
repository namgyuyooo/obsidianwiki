# Paperclip UX/UI User Intent Analysis

Paperclip 화면의 핵심 역할은 `작업 실행기`를 보여주는 것이 아니라 `지금 어떤 읽기/검수 작업이 필요하고, 그것을 안전하게 실행해 결과를 채택할지`를 빠르게 판단하게 만드는 것이다.

## 한줄 결론

현재 Paperclip 화면은 `템플릿 선택과 실행`은 가능하지만, 사용자가 실제로 먼저 알고 싶은 `왜 지금 이 작업을 해야 하는지`, `지금 실행해도 안전한지`, `결과를 어떻게 채택할지`를 전면에 두는 구조는 아직 약하다.

## 이 화면의 실제 사용자

### 1. 운영자

- 지금 어떤 수집/검수 요청을 넣어야 하는지 판단한다.
- 잘못된 실행이나 과잉 실행을 피하고 싶다.
- 결과를 읽고 위키 승격 여부를 빠르게 결정하고 싶다.

### 2. 검토자

- 이미 누가 만든 task가 안전한지 본다.
- 승인/보류/재실행 기준을 분명히 보고 싶다.
- 결과물이 실제로 쓸 만한지 확인하고 싶다.

### 3. GLM/챗 사용자

- 챗에서 제안된 작업이 실제로 어떤 queue와 결과를 만들었는지 연결해 보고 싶다.
- 템플릿 이름보다 `어떤 문제를 해결하는 요청인지`를 이해하고 싶다.

## 핵심 목적과 의도

### 최상위 목적

- 위키 운영 중 생기는 `읽기`, `검수`, `보조 실행`을 안전하게 통제한다.
- 실행 결과를 `다운로드`, `지식 승격`, `보류` 중 하나로 빠르게 처리한다.

### 세션 단위 의도

1. 지금 Paperclip이 필요한 상황인지 먼저 판단한다.
2. 필요한 경우 가장 맞는 작업 종류를 빠르게 찾는다.
3. 실수 없이 최소 입력으로 요청을 만든다.
4. 대기열과 이벤트를 보며 실행 위험을 관리한다.
5. 나온 결과를 바로 읽고 다음 행동으로 넘긴다.

### 사용자가 화면에서 실제로 묻는 질문

- `지금 내가 뭘 해야 하지?`
- `이 요청을 지금 돌려도 안전한가?`
- `어떤 템플릿을 골라야 하지?`
- `이미 돌고 있는 작업과 충돌하지 않나?`
- `방금 나온 결과를 위키로 올릴 가치가 있나?`

## 현재 화면과 사용자 의도의 어긋남

### 1. 첫 시선이 `무엇을 해결할지`보다 `어떤 템플릿을 돌릴지`에 고정된다

- 현재 메인 카드가 바로 `읽기 요청 만들기`다.
- 사용자는 템플릿 id나 command보다 먼저 `지금 필요한 작업 유형`을 알고 싶다.
- 결과적으로 초심자는 `왜 이 템플릿을 골라야 하는지`를 화면 안에서 학습하기 어렵다.

### 2. queue와 result는 보이지만 `다음 권장 행동`이 약하다

- `queue`, `result`, `event`는 노출되지만 각 상태에서 무엇을 해야 하는지 직접 말해주지 않는다.
- 예: `queue pressure`는 보여도 `지금 새 요청보다 기존 task 검토를 먼저 하라`는 식의 행동 권고가 없다.

### 3. 안전성은 표시되지만 승인 맥락은 얕다

- `READY/OFFLINE`, `테스트 기본/실행 기본`은 보인다.
- 하지만 `이 task는 승인 후 실행`, `이 결과는 검토 후 승격`, `이 요청은 기존 queue와 중복 가능` 같은 운영 문맥은 약하다.

### 4. 완료 결과는 읽을 수 있지만 `채택 판정 흐름`이 약하다

- 현재 결과 영역은 `열기`, `다운로드`, `지식승격` 중심이다.
- 그러나 실제 사용자 판단은 `채택`, `보류`, `폐기`, `재실행 요청`으로 더 나뉜다.
- 즉 결과 화면이 `viewer`에는 가깝지만 `review inbox`로는 아직 부족하다.

### 5. 언어 레이어가 혼합돼 인지 부하가 남는다

- `bridge`, `request`, `queue pressure`, `result` 같은 영어 운영어와 한국어 설명이 혼재한다.
- 이 화면의 실사용자는 운영 판단을 빨리 내려야 하므로, 상태 라벨은 한국어 기준으로 더 즉시 읽혀야 한다.

## 재정의해야 할 화면 원칙

### 원칙 1. 템플릿보다 사용자 의도를 먼저 보여준다

- 템플릿 라이브러리는 필요하지만 첫 질문은 `무슨 작업을 만들까`가 아니라 `무슨 문제를 해결할까`여야 한다.

### 원칙 2. 실행보다 판단을 먼저 돕는다

- Paperclip은 상위 control plane이므로, 실행 버튼보다 `지금 실행해도 되는가`를 먼저 보여야 한다.

### 원칙 3. 결과는 문서 미리보기가 아니라 판정 inbox로 다룬다

- 결과물은 읽고 끝나는 것이 아니라 위키 채택 여부를 결정하는 운영 객체다.

### 원칙 4. 한 화면 안에서 `요청 -> 진행 -> 결과` 전체 흐름을 닫는다

- 사용자가 다른 탭으로 이동하지 않고도 요청 생성, 위험 확인, 결과 채택까지 이어져야 한다.

## 권장 정보 구조

### 상단: 현재 권장 행동

- `지금 할 일 1개`를 가장 크게 보여준다.
- 예:
  - `대기열이 비어 있어 새 읽기 요청 생성 가능`
  - `실행 중 task 3개, 신규 요청보다 결과 검토 우선`
  - `오프라인 상태, bridge 복구 전 실행 금지`

### 좌측: 의도 중심 작업 라이브러리

템플릿 대신 아래 같은 사용자 의도 그룹으로 먼저 묶는다.

- `문서 읽기 요청`
- `검수/검증 요청`
- `기존 결과 재검토`
- `승격 후보 만들기`

각 항목 안에 실제 template, command, safety mode를 2차 정보로 넣는다.

### 중앙: 요청 작성기

- 요청 이름
- 자연어 지시
- 프로젝트 힌트
- 예상 산출물
- 안전 모드
- 실행 전 체크포인트

중앙 영역의 목표는 `빠른 입력`이 아니라 `실수 없는 입력`이다.

### 우측: 안전/승인 레일

- bridge 상태
- dry-run / run 구분
- queue 충돌 가능성
- 승인 필요 여부
- 최근 실패 원인
- 지금 실행 시 주의사항

### 하단: 결과 inbox

결과 카드는 단순 완료 목록이 아니라 아래 상태를 가져야 한다.

- `검토 필요`
- `승격 가능`
- `보류`
- `재실행 필요`
- `실패 원인 확인`

## 구체 UI 개선안

### P0. 첫 화면의 질문을 바꾼다

- `읽기 요청 만들기` 단일 헤더를 `지금 필요한 작업` 중심으로 바꾼다.
- 첫 카드에서 바로 template를 고르게 하지 말고, `무엇을 하려는가` preset을 먼저 보여준다.

### P0. 권장 다음 행동 배너를 추가한다

- queue, bridge, result 상태를 조합해 가장 먼저 해야 할 행동을 1줄로 제안한다.
- 이 배너가 없으면 사용자는 숫자를 읽고 직접 해석해야 한다.

### P0. 결과 카드를 `판정 큐`로 재설계한다

- `다운로드/지식승격` 외에 `보류`, `재실행 요청`, `무시`를 추가한다.
- 결과를 채택 대상으로 다루면 운영 흐름이 훨씬 명확해진다.

### P1. 작업 종류를 `template 목록`이 아니라 `의도별 그룹`으로 묶는다

- 초심자는 command 중심 목록에서 길을 잃기 쉽다.
- 고급 사용자는 그룹 안에서 command와 safety를 보게 하면 된다.

### P1. queue item에 실무 판단 정보를 늘린다

- 현재 status/command 외에 `project hint`, `생성 이유`, `승인 상태`, `중복 가능성`, `다음 액션`을 함께 보여준다.

### P1. 상태 언어를 한국어 기준으로 정리한다

- `bridge`, `queue pressure`, `result` 같은 라벨을 바로 이해되는 운영 한국어로 통일한다.
- 영어는 command/path 같은 2차 기술정보로만 남긴다.

### P2. 채팅과의 연결을 더 명시한다

- 이 task가 어느 chat 제안에서 왔는지, 어떤 skill tag와 연결되는지 보여준다.
- 사용자는 `왜 이 작업이 생겼는지`를 되짚을 수 있어야 한다.

## 추천 마이크로카피

- `읽기 요청 만들기` -> `지금 필요한 작업 만들기`
- `요청만 저장` -> `대기열에 넣기`
- `바로 실행` -> `검토 후 실행 요청`
- `완료 결과` -> `검토할 결과`
- `지식승격` -> `위키 승격 후보 만들기`
- `queue pressure` -> `대기열 상태`

## 성공 판단 기준

- 사용자가 첫 10초 안에 `지금 해야 할 다음 행동`을 이해한다.
- 신규 사용자가 template id를 몰라도 올바른 작업 생성 경로를 찾는다.
- 결과 화면에서 `읽고 끝남`보다 `채택/보류/재실행` 판단 비율이 높아진다.
- 잘못된 실행보다 `보류 후 검토`가 늘어난다.

## 바로 반영할 구현 우선순위

1. `추천 다음 행동` 계산 로직 추가
2. template 목록을 의도 그룹 기반 섹션으로 재배치
3. 결과 리스트에 review state 추가
4. queue item 메타데이터 확장
5. 한국어 상태 라벨 정리
