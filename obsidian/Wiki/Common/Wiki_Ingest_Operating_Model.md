# Wiki Ingest Operating Model

위키는 “인제스트 저장소”가 아니라 “실무 운영 시스템”으로 운영합니다.
문서를 읽고 끝내지 않고, 실제 판단과 다음 행동까지 연결되도록 설계합니다.

## 공간 유형별 운영 단위

위키의 top-level 공간은 아래 4종으로 나눕니다.

- `project`
  - 독립 실행 단위
  - 증적, 결정, 액션, 리스크, L1 메모리까지 함께 관리
- `account`
  - 고객/계정 상위 현황판
  - 여러 프로젝트의 관계, 계정 차원의 상업 이슈, 다음 접점을 관리
- `common`
  - 운영 모델, 자동화, 인덱스, 거버넌스, 제도 문서
- `shared`
  - 프로젝트에서 승격된 재사용 자산

같은 템플릿을 모든 공간에 강제하지 않습니다.
특히 `account`, `common`, `shared`는 프로젝트용 evidence 세트를 기계적으로 복제하지 않습니다.

## 실무 운영의 기본 질문

모든 인제스트와 위키 수정은 아래 다섯 질문 중 하나 이상에 답해야 합니다.

1. 무엇이 바뀌었는가
2. 무엇이 확정되었는가
3. 무엇이 아직 충돌하거나 미확정인가
4. 다음 액션은 무엇이고 누구의 일인가
5. 그 판단을 뒷받침하는 근거는 어디 있는가

이 다섯 질문을 더 선명하게 만들지 못하는 편집은 운영 가치가 낮습니다.

## 1. 참조 보존 계층

모든 내용을 위키 본문에 복제하지 않고, 어디를 다시 열어봐야 하는지 추적하는 층입니다.

- 보관 대상
  - Slack 링크
  - 웹 링크
  - Google Drive 링크
  - 로컬 파일 경로
  - 링크를 남기기 어려울 때의 파일명 fallback
  - 이 참조가 어디에 설명되어 있는지에 대한 위키 링크
  - 접근 메모와 읽기 상태
- 대표 문서
  - `Reference_Register.md`

## 2. 원문 보존 계층

문서에서 실제로 읽은 내용을 보존하는 층입니다.

- 보관 대상
  - 문서명
  - 날짜
  - 문서 유형
  - 핵심 발췌
  - 수치 원문
  - 결정사항 원문
  - 충돌 원문
  - 출처 위치
- 추천 문서
  - `Sources.md`
  - `Evidence_Log.md`
  - `Document Notes/문서명.md`

## 3. 정제 지식 계층

원문을 프로젝트 실행과 재사용에 맞게 구조화하는 층입니다.

- 대표 문서
  - `Project_Overview.md`
  - `KPI.md`
  - `Decisions.md`
  - `Risks.md`
  - `Market Analysis.md`
  - `Revenue Model.md`
  - `Equipment.md`
  - `Architecture.md`

## 4. 이력 및 변경 관리 계층

무엇이 언제 어떻게 바뀌었는지 추적하는 층입니다.

- 보관 대상
  - 날짜형 업데이트 블록
  - 충돌 등록
  - 이전 안 대비 변경사항
  - 확정/미확정 구분
- 추천 문서
  - `Change_Log.md`
  - `Conflict_Register.md`
  - 기존 문서 하단 업데이트 블록

## 5. 실행 현황 계층

실무자가 위키를 열었을 때 가장 먼저 보는 층입니다.

- 보관 대상
  - 현재 상태
  - 현재 단계
  - 마지막 의미 있는 변화
  - 막힌 이슈 / 미확정 / 충돌
  - 다음 액션
  - 연결 근거
- 대표 문서
  - `Status.md`
  - `hub.md`
  - `Action_Items.md`
  - `Decisions.md`
  - `Risks.md`
  - `obsidian/L1_memory/{ProjectName}.md`

## 신규 이벤트 처리 순서

새로운 업무 이벤트가 생기면 바로 정제 문서에 결론을 쓰지 않고 아래 순서로 승격합니다.

1. 원본 이벤트를 캡처합니다.
2. 관련 파일, 메모, Drive 경로, Slack 링크, 웹 링크를 `Reference_Register.md`에 링크 우선으로 등록합니다.
3. 링크를 안정적으로 남기기 어렵다면 `Sources.md`에 파일명, 경로, 접근 메모를 fallback으로 기록합니다.
4. 실제 문장, 수치, 결정사항, 제약 조건을 `Evidence_Log.md`에 추출합니다.
5. 기존 값과 다른 수치, 일정, 요구사항은 `Conflict_Register.md`에 보류 상태로 등록합니다.
6. 실제 위키 문서나 보고서를 수정했으면 `Change_Log.md`에 기록합니다.
7. `Action_Items.md`, `Decisions.md`, `Risks.md`에 실무 판단과 다음 행동을 반영합니다.
8. `Status.md`에 상태 라벨, 단계, 헬스, 오너, 막힘, 다음 게이트를 갱신합니다.
9. 허브 상단의 실행 현황판, 막힘/충돌, 다음 액션을 갱신합니다.
10. 프로젝트 상태, 핵심 결정, 미해결 이슈가 바뀌면 `obsidian/L1_memory/{ProjectName}.md`를 갱신합니다.

개인 기록은 이 업무 위키에 저장하지 않습니다. 개인/업무가 섞인 이벤트는 업무상 필요한 사실만 프로젝트 문서에 남기고, 개인 맥락은 별도 개인 공간에 둡니다.

## 운영 원칙

- 위키는 구조만 관리하지 말고 실제 내용도 관리합니다.
- 로컬 위키만 보지 말고 연결된 Google Drive도 함께 탐색합니다.
- Drive 탐색은 접근 가능한 모든 Drive와 Shared Drive를 샅샅이 확인하는 것을 기본값으로 합니다.
- 문서 탐색 시 파일명뿐 아니라 상위 폴더명도 함께 판단합니다.
- 파일명보다 폴더명이 더 강한 맥락을 줄 수 있으므로 Shared Drive 이름, 상위 폴더명, 상상위 폴더명을 함께 기록합니다.
- 상위 폴더명에 `RTM`이 포함되면 1차 분석 후보로 포함합니다.
- `RTM`이 없더라도 파일명 또는 문서 내용이 프로젝트 핵심 키워드와 강하게 일치하면 후보에 포함합니다.
- `hwp`와 `hwpx`는 예외 없이 조사 대상에 포함합니다.
- `hwp`와 `hwpx`는 필요 시 `rhwp` 기반 점검 절차를 사용해 파일 정보, 구조 덤프, 페이지 구조, SVG 렌더 결과를 확인합니다.
- 모든 내용을 위키 본문에 복제하지 않습니다.
- 먼저 `Reference_Register.md`에 다시 찾아갈 링크와 fallback 파일명을 남기고, 위키 본문에는 실행 판단에 필요한 요약만 남깁니다.
- 요약만 저장하지 않고, 핵심 문장과 근거를 함께 남깁니다.
- 해석과 원문 근거를 분리합니다.
- 숫자는 반드시 문서명과 날짜를 함께 기록합니다.
- 상충되는 수치는 둘 다 남기고 충돌 상태를 유지합니다.
- 다만 `정합성 확인 필요`, `검토 필요`, `허브 보강 필요` 같은 약한 문장은 자동으로 conflict로 승격하지 않습니다.
- 명시적 상충값이 없다면 `Action_Items.md`, `Status.md`, `Risks.md`, `Decisions.md`, `hub.md` 중 더 실무적인 목적지로 보내는 편을 우선합니다.
- 기존 문서를 덮어쓰지 말고 날짜형 업데이트 블록으로 append 합니다.
- 나중에 사업계획서, 제안서, 보고서, 발표자료, 영문 문서로 전환 가능한 구조를 우선합니다.
- 허브 상단만 읽어도 `상태`, `막힘`, `다음 액션`, `근거`를 파악할 수 있어야 합니다.
- `Change_Log.md`는 위키 변경 이력이고, 허브는 실제 업무 추진 현황판입니다. 둘을 섞지 않습니다.
- 문서 정리가 아니라 실무 의사결정을 돕는 방향으로 우선순위를 둡니다.
- 따라서 에이전트는 가능하면 `무엇이 문제인지`뿐 아니라 `어느 문서를 어떻게 수정하면 좋은지`까지 능동적으로 제안하거나 직접 반영해야 합니다.

## 사업계획서 계열 대표본 선별 원칙

- 사업계획서, 제안서, 투자자료, 보고서, 발표자료를 읽을 때는 먼저 관련 후보 문서를 수집합니다.
- 관련 후보 수집은 특정 폴더에 한정하지 않고 접근 가능한 모든 Drive를 대상으로 합니다.
- 같은 계열 문서가 여러 개 있으면 `Primary`, `Secondary`, `Hold`로 구분합니다.
- 대표본 선정 시 아래 순서를 함께 봅니다.
  - 상위 폴더에 `RTM` 포함 여부
  - Shared Drive 이름과 폴더 계층 맥락
  - 파일명에 `vf`, `최종`, `최종본`, `final`, `final본`, `확정`, `제출본`, `발표본` 포함 여부
  - 파일명 마지막 버전번호
  - 마지막 수정일
  - 프로젝트 핵심 키워드 일치도
  - 문서 유형 적합성
  - 문서 내용의 완결성
- 파일명, 버전번호, 수정일 중 하나만으로 대표본을 확정하지 않습니다.
- `hwp`와 `hwpx`를 형식상 이유로 후보군에서 제외하지 않습니다.
- 대표본을 먼저 읽고, 비교본을 읽어 변경사항을 `Change_Log.md`와 `Conflict_Register.md`에 기록합니다.

## 허브 운영 기준

`project` 허브는 아래 순서를 기본으로 둡니다.

1. `운영 메모`
2. `실행 현황판`
3. `현재 막힘 / 충돌`
4. `다음 액션`
5. `최근 업데이트`
6. `참조 링크`
7. `증적/근거 링크`
8. `운영 링크`

`account` 허브는 아래를 기본으로 둡니다.

1. `계정 운영 메모`
2. `활성 프로젝트`
3. `계정 차원 막힘 / 에스컬레이션`
4. `다음 접점`
5. `프로젝트 관계 링크`

`common` 허브는 아래를 기본으로 둡니다.

1. `운영 메모`
2. `활성 운영 모델`
3. `승격/정비 큐`
4. `자동화/거버넌스 진입점`
5. `핵심 운영 링크`

`shared` 허브는 아래를 기본으로 둡니다.

1. `공유 자산 메모`
2. `재사용 자산 목록`
3. `적용 대상`
4. `승격 출처`
5. `연결 링크`
