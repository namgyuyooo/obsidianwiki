Worker가 카메라를 찾고 연결한다. - 카메라 파라미터를 초기화한다. - 스트리밍 준비가 완료된다. - Orchestrator가 전체 준비 상태를 파악한다.  ## 예외사항  - 카메라가 연결되지 않는다. - 동일 ID의 카메라가 둘 이상 잡히거나, 기대한 카메라가 다른 물리 장비로 연결된다. - 카메라 연결은 되지만 스트리밍 시작에 실패한다. - 특정 파라미터(exposure/gain/ROI/trigger mode)가 해당 SDK/장비에서 지원되지 않는다. - 파라미터 적용 중 일부만 반영되고 일부는 실패한다. - 초기화 시간 초과가 발생한다. - 카메라 연결은 됐지만 실제 프레임을 받지 못한다. - 다중 카메라일 때 일부만 ready 상태가 된다. - 시간 동기화가 필요한데 아직 공통 시간축 확보 전이다.  ## 아키텍처 관점 점검 포인트  ### 책임 귀속  - Worker 수준 오류와 Orchestrator 수준 오류의 경계가 선언되어 있는가 - 장치 능력(capabilities) 선언 경로가 포트 계약에 포함되어 있는가 - 일부 카메라 미준비 상태의 전체 상태 집계 책임이 Orchestrator에 귀속되어 있는가 - 초기화 타임아웃의 감지·보고 책임이 Worker에 귀속되어 있는가 - 카메라 논리 식별자와 물리 장비 매핑 책임 위치가 선언되어 있는가  ### 실패 모드·반응 경로  - 초기화 타임아웃 발생 시 Orchestrator의 반응 경로(세션 시작 차단, 다른 Worker 중단 등)가 명시되어 있는가 - 일부 Worker만 ready 상태일 때 수집 시작 여부 결정권이 귀속되어 있는가 - 파라미터 적용 부분 실패의 감지·보고 경로가 정의되어 있는가 - 동일 ID 중복 또는 잘못된 물리 매핑의 감지 경로가 정의되어 있는가  ---  # 검사 전 메타데이터 준비 단계  ## 정상 흐름  - 세션 시작 전 필요한 외부 정보와 사용자 입력 정보가 준비된다. - Session Manager가 세션에 연결할 정보를 확보한다.  ## 예외사항  - MES/상위 시스템 연결 실패로 작업지시 메타데이터를 가져오지 못한다. - PLC/상위 설비 신호는 오지만 제품/lot 정보는 없다. - 사용자 입력이 누락되거나 잘못 입력된다. - 외부 시스템 정보와 사용자 입력 정보가 서로 충돌한다. - 메타데이터가 늦게 들어와 촬영 시작 시점에 준비되지 않는다. - 외부 시스템 메타데이터가 변경되었는데 이미 열린 세션과 불일치한다. - source of truth가 불명확하여 어느 값을 기록해야 할지 모른다.  ## 아키텍처 관점 점검 포인트  ### 책임 귀속  - 메타데이터 4분류(External·User Input·Runtime Context·Derived)의 관리 책임이 Session Manager에 귀속되어 있는가 - 필수/선택 메타데이터 구분 책임이 Session Manager에 귀속되어 있는가 - 메타데이터 출처 간 충돌 중재 책임이 Session Manager에 귀속되어 있는가  ### 실패 모드·반응 경로  - 필수 메타 미준비 상태에서 세션 시작 차단·허용 결정권이 명시되어 있는가 - 출처 간 충돌 발생 시 적용되는 우선순위 규칙이 규약 수준에서 정의되어 있는가 - 세션 개시 후 외부 메타가 갱신된 경우의 처리 경로(무시·반영·기록)가 정의되어 있는가 - 외부 시스템 연결 실패 시 선택 메타로 강등 또는 시작 차단 결정 경로가 정의되어 있는가  ---  # 검사 시작 감지 단계  ## 정상 흐름  - Trigger Detector가 시작 신호를 감지한다. - 현재 시스템 상태를 반영해 유효한 시작 이벤트인지 판단한다. - Orchestrator에 시작 트리거를 전달한다.  ## 예외사항  - 시작 신호가 들어오지 않는다. - 시작 신호가 중복 발생한다. - 시작 신호와 종료 신호가 비정상 순서로 들어온다. - OCR이 잘못 읽어 false start가 발생한다. - Start mark가 오검출된다. - Start mark가 있어야 하는데 조명/속도 문제로 검출되지 않는다. - 수동 시작과 자동 시작이 동시에 들어온다. - 한 세션이 아직 닫히지 않았는데 다시 시작 신호가 들어온다. - 다중 트리거 소스가 서로 상충한다. - 시간 동기화 미확보 상태에서 시작 신호가 들어온다.  ## 아키텍처 관점 점검 포인트  ### 책임 귀속  - Trigger Detector의 중복 제거·상태 기반 필터링 책임이 선언되어 있는가 - 복수 트리거 입력의 중재 책임이 Trigger Detector 내부에 귀속되어 있는가 - 시간 동기화 미확보 시 세션 시작 차단 결정권이 특정 컴포넌트(Orchestrator 등)에 귀속되어 있는가 - 세션 미종료 상태에서의 재시작 차단 결정권이 귀속되어 있는가  ### 실패 모드·반응 경로  - 시작·종료 비정상 순서 신호의 무효화 경로가 정의되어 있는가 - 수동 시작과 자동 시작 동시 입력 시의 우선순위 규칙이 선언되어 있는가 - false start(OCR 오독, Start mark 오검출)를 이후 단계에서 정정하는 경로가 정의되어 있는가 - 시작 신호 장기 부재 시 상위 보고 경로가 정의되어 있는가  ---  # 프레임 수집 단계  ## 정상 흐름  - Worker가 카메라로부터 프레임을 취득한다. - Orchestrator가 전체 수집 상태를 모니터링한다. - Session Manager가 세션 스코프를 유지한다.  ## 예외사항  - 프레임 드랍이 발생한다. - SDK 콜백은 오지만 payload가 비정상이다. - 프레임 상태가 incomplete다. - 특정 시점부터 프레임이 멈춘다. - 카메라 연결은 살아 있지만 실제 데이터가 안 들어온다. - FPS가 기대치보다 낮다. - 광량 부족으로 노출이 부족하다. - blur로 불량이 판독 불가하다. - 카메라 내부 버퍼가 꽉 찬다. - Worker 내부 큐가 넘친다. - 다중 카메라에서 일부 카메라만 계속 프레임 누락이 난다. - 동기화된 프레임이라 기대했는데 실제 시간축이 어긋난다. - 장치 재연결이 필요하지만 세션 도중 발생한다.  ## 아키텍처 관점 점검 포인트  ### 책임 귀속  - Worker의 프레임 취득 실패 감지·보고 책임이 선언되어 있는가 - Orchestrator의 degraded 상태 인지 경로가 명시되어 있는가 - 프레임 유실·지연의 관측 기록 위치(세션 메타·시스템 로그)가 선언되어 있는가 - Acquisition Parameter Manager의 런타임 파라미터 재조정 지시 경로가 수집 흐름과 분리되어 있는가  ### 실패 모드·반응 경로  - SDK 콜백 payload 비정상 또는 incomplete 프레임의 Adapter 경계 도메인 에러 번역 경로가 정의되어 있는가 - 카메라 재연결 필요 시의 대응 정책 결정권(중단 · 재연결 시도 · degraded 진행)이 귀속되어 있는가 - Worker 내부 큐 오버플로우 시의 손실 기록 경로가 정의되어 있는가 - 프레임 장기 부재(연결은 살아 있으나 데이터 없음) 감지·보고 경로가 정의되어 있는가 - 다중 카메라에서 특정 카메라 드랍률만 비정상적으로 높을 때 degraded 선언 경로가 정의되어 있는가  ---  # 프레임 저장 및 세션 스테이징 단계  ## 정상 흐름  - Worker가 취득 직후 프레임을 세션 저장 경계에 기록한다. - 최소 메타데이터가 함께 기록된다. - Session Manager는 세션 수준 일관성을 관리한다.  ## 예외사항  - 저장 디스크 용량 부족 - 디스크 쓰기 속도 부족 - 파일 생성 실패 - 특정 프레임 저장 실패 후 다음 프레임은 들어온다 - 저장 지연이 누적되어 수집에 역류한다 - 파일명/경로 충돌 - 세션 스테이지 디렉터리 생성 실패 - 프레임은 저장됐지만 메타 기록이 누락된다 - 메타는 기록됐지만 프레임 파일이 없다 - 프로세스 비정상 종료로 일부 프레임만 저장된다 - 세션 중간에 머신 재부팅 또는 앱 크래시 발생 - 다중 카메라에서 카메라별 기록 개수가 맞지 않는다  ## 아키텍처 관점 점검 포인트  ### 책임 귀속  - Worker의 저장 내구성 책임이 선언되어 있는가 - Worker의 쓰기 대상을 로컬 저장 경계로 한정하는 원칙이 아키텍처 수준에서 선언되어 있는가 - 저장 용량 부족의 감지·보고 책임이 귀속되어 있는가 - Session Manager의 프레임/메타 불일치 식별 책임이 선언되어 있는가 - 비정상 종료 후 회복 가능한 세션 경계 유지의 책임이 귀속되어 있는가  ### 실패 모드·반응 경로  - 파일 생성 실패·경로 충돌 시 Worker의 실패 분류·보고 경로가 정의되어 있는가 - 저장 병목이 누적되어 수집 경로에 역류하려 할 때의 완화 구조가 명시되어 있는가 - 저장 용량 부족 보고 수신 시 Orchestrator의 반응 경로(세션 시작 차단·진행 중 세션 강제 종료)가 정의되어 있는가 - 프레임 ↔ 메타 불일치 발견 시의 세션 상태 표현 방식이 정의되어 있는가 - 프로세스 비정상 종료 또는 머신 재부팅 후 재기동 시 미완료 세션의 복구 책임이 귀속되어 있는가 - 다중 카메라 환경에서 카메라별 기록 개수 불일치의 감지·보고 경로가 정의되어 있는가  ---  # 검사 종료 감지 단계  ## 정상 흐름  - Trigger Detector가 종료 신호를 감지한다. - Orchestrator가 수집 종료를 지시한다. - Worker가 종료를 완료한다.  ## 예외사항  - 종료 신호가 오지 않는다. - 종료 신호가 너무 빨리 온다. - 종료 신호가 중복된다. - 시작 없이 종료만 온다. - 자동 종료가 실패해 세션이 계속 열린다. - 일부 카메라는 종료됐는데 일부는 계속 수집 중이다. - 타임아웃에 의해 강제 종료가 필요하다. - 운영자가 수동 종료를 눌렀지만 카메라는 계속 동작한다.  ## 아키텍처 관점 점검 포인트  ### 책임 귀속  - 종료 개시 책임의 이중 귀속(Trigger Detector 정상 경로, Orchestrator 강제 경로)이 선언되어 있는가 - 정상 종료·강제 종료의 유형 구분이 세션 생명주기에 선언되어 있는가 - 부분 종료 상태의 감지·수습 책임이 귀속되어 있는가  ### 실패 모드·반응 경로  - 종료 신호 장기 부재(타임아웃) 시 강제 종료 전환 결정권이 명시되어 있는가 - 수동 종료 명령 이후 카메라 계속 동작 시의 강제 정지 경로가 정의되어 있는가 - 시작 없이 종료만 도달한 경우의 무효화 경로가 정의되어 있는가 - 너무 이른 종료 신호(최소 세션 길이 미달)의 정책적 처리 경로가 정의되어 있는가  ---  # 세션 종료 및 Finalize 단계  ## 정상 흐름  - Session Manager가 세션을 닫는다. - 세션 수준 결과와 메타데이터를 정리한다. - 학습용 인계 가능한 상태가 된다.  ## 예외사항  - 세션 종료는 됐지만 일부 프레임이 아직 flush되지 않았다. - 세션 메타 확정 중 실패한다. - Derived metadata 계산 중 실패한다. - 종료 후 세션 상태와 실제 파일 상태가 불일치한다. - 강제 종료 세션을 정상 세션처럼 표시해버린다. - 다중 카메라 세션에서 일부 카메라 결과만 finalize된다. - 종료 직후 재기동 시 미완료 세션 복구가 필요하다.  ## 아키텍처 관점 점검 포인트  ### 책임 귀속  - Session Manager의 정상·비정상 세션 구분 관리 책임이 선언되어 있는가 - Finalize 실패의 세션 상태 표현 책임이 선언되어 있는가 - 세션 close와 프레임 flush 완료 간 동기화 책임이 명확히 귀속되어 있는가  ### 실패 모드·반응 경로  - Derived metadata 계산 실패 시 세션 상태 표현 방식이 정의되어 있는가 - 재기동 시 미완료 세션 복구 절차의 개시 책임이 명시되어 있는가 - 다중 카메라 환경에서 일부 카메라만 finalize된 상태의 수습 경로가 정의되어 있는가 - 강제 종료 세션이 정상 세션으로 잘못 표시되지 않도록 하는 기록 경로가 정의되어 있는가  ---  # 사후 검증 및 학습용 데이터 인계 단계  ## 정상 흐름  - 수집된 세션을 사람이 검토할 수 있다. - 학습용 데이터셋 후보로 인계할 수 있다.  ## 예외사항  - 실제 불량 제품을 찍었는데 불량이 관측되지 않는다. - 불량은 관측됐지만 해상도/조명/blur 때문에 판독이 어렵다. - 불량이 너무 작아 프레임 사이로 유실된다. - 실제로는 정상 제품이었는데 불량 샘플로 잘못 분류됐다. - 불량 위치를 알고 있었는데 세션 결과와 매칭이 안 된다. - 동일 조건 반복 테스트에서 결과가 달라 재현성이 없다. - 광학 조건 변경 후 이전 세팅과 비교가 안 된다. - 학습용으로 충분한 메타데이터가 없어서 데이터 활용성이 떨어진다.  ## 아키텍처 관점 점검 포인트  ### 책임 귀속  - Session Manager의 세션/메타/이미지 묶음 제공 책임이 선언되어 있는가 - 운영/검수 시스템이 연결될 확장 지점이 명시되어 있는가 - 튜닝/레시피 관리 기능이 추가될 확장 지점이 명시되어 있는가 - 세션 산출물이 read-only로 인계된다는 원칙이 선언되어 있는가  ### 실패 모드·반응 경로  - 외부 소비자(추론·검수 시스템)의 비정상 접근이 세션 상태에 영향을 주지 않는 구조가 명시되어 있는가 - 광학 조건 변경 전후 세션 비교를 위한 설정 이력 추적 경로가 정의되어 있는가 - 메타데이터 부족으로 인한 활용성 저하를 사전에 감지할 수 있는 경로가 정의되어 있는가  ---  # 튜닝 및 재세팅 관점 예외  ## 정상 흐름  - 기준 불량 샘플로 반복 테스트를 수행한다. - 광학 조건과 시스템 내부 수집 조건을 조절한다. - 재현성과 최소 검출 조건을 만족하는 세팅을 찾는다.  ## 예외사항  - 광학 조건 변경 후 시스템 내부 파라미터를 갱신하지 않았다. - 노출/게인/FPS/ROI 조합이 제품 조건과 맞지 않는다. - 카메라 SDK가 원하는 파라미터를 지원하지 않는다. - 수동 튜닝은 가능하지만 저장/재적용이 불가능하다. - 세팅값 변경 이력이 남지 않는다. - 제품 종류가 바뀌어도 이전 세팅을 그대로 사용한다. - 테스트용 세팅과 운영용 세팅이 혼용된다.  ## 아키텍처 관점 점검 포인트  ### 책임 귀속  - 튜닝/레시피 관리 계층의 추가 확장 지점이 아키텍처에 명시되어 있는가 - Adapter의 설정 적용 책임과 세팅 판단 책임의 분리 원칙이 선언되어 있는가 - 세팅 변경 이력의 세션 메타 연결 경로가 선언되어 있는가 - 파라미터 프로파일의 저장·재적용 책임이 귀속되어 있는가  ### 실패 모드·반응 경로  - 특정 파라미터가 Adapter 능력 선언에 포함되지 않을 때 상위의 적용 시도 차단 경로가 정의되어 있는가 - 테스트용 프로파일과 운영용 프로파일의 혼용 방지 경로(프로파일 구분 태그 등)가 정의되어 있는가 - 광학 조건만 변경되고 프로파일이 갱신되지 않은 상태의 감지·경고 경로가 정의되어 있는가  ---  # 다중 카메라 확장 시 추가 예외  ## 정상 흐름  - 여러 Worker가 동시에 동작한다. - Orchestrator가 전체 상태를 집계한다. - Session Manager가 카메라별 산출물을 일관되게 관리한다.  ## 예외사항  - 일부 카메라만 준비 완료 - 일부 카메라만 시작/종료 성공 - 카메라 간 시간축 불일치 - 동일 세션 내 카메라별 프레임 수 심각한 차이 - 특정 카메라의 드랍률만 비정상적으로 높음 - 카메라별 광학 조건이 달라 시인성이 다름 - 한 카메라 장애가 전체 세션을 중단해야 하는지 불명확 - 일부 카메라 degraded 상태에서 세션 계속 수행 여부가 모호  ## 아키텍처 관점 점검 포인트  ### 책임 귀속  - Orchestrator의 전역 상태 표현 책임이 선언되어 있는가 - 부분 장애 사실의 세션 메타 기록 책임이 Session Manager에 귀속되어 있는가 - 시간 동기화 실패의 신뢰도 상태 표현 책임이 귀속되어 있는가  ### 실패 모드·반응 경로  - 한 카메라 장애 시 전체 세션 중단 또는 degraded 진행 결정권이 귀속되어 있는가 - 카메라 간 프레임 수 심각한 차이의 감지·보고 경로가 정의되어 있는가 - 공통 시간축 이탈 구간의 프레임을 정상 데이터와 분리해 표시하는 기록 경로가 정의되어 있는가  ---  # 관측 가능성 (횡단)  ## 정상 흐름  - 시스템 상태와 세션 진행을 외부 관찰자가 확인할 수 있다. - 오류·손실·경고 이벤트가 사후 분석 가능한 형태로 기록된다. - 성능 지표(FPS·드랍률·지연·저장 속도)가 운영 중 확인 가능하다.  ## 예외사항  - 로그 저장 실패가 수집 흐름에 영향을 준다. - 세션 실패 원인을 사후에 추적할 수 없다. - 다중 카메라 중 특정 카메라만 degraded일 때 구분이 안 된다. - 메트릭 수집기가 부재할 때 시스템이 동작하지 않는다. - 오류 분류가 불명확해 설정·환경·장비 중 어느 문제인지 판별이 어렵다.  ## 아키텍처 관점 점검 포인트  ### 책임 귀속  - 프레임 유실·지연의 기록 위치(세션 메타 vs 시스템 로그)가 선언되어 있는가 - 컴포넌트별 상태(Worker 준비/수집/종료, Orchestrator 집계 상태, Session Manager 세션 상태)의 공표 경로가 명시되어 있는가 - 오류 유형 분류 체계(설정·환경·장비·세션·운영)의 보고 경로가 선언되어 있는가 - 관측 데이터의 세션 메타 결합 방식이 귀속되어 있는가  ### 실패 모드·반응 경로  - 로그·메트릭 쓰기 실패가 수집 흐름으로 역류하지 않는 구조가 명시되어 있는가 - 메트릭 수집 컴포넌트 부재 시 시스템 기능이 계속 동작하는 구조가 선언되어 있는가 - 관측 데이터 누락 시의 감지 경로가 정의되어 있는가  ---  # 설정 관리 (횡단)  ## 정상 흐름  - 설정 파일로부터 Adapter 선택·파라미터 프로파일·임계값이 로드된다. - 설정 변경이 세션 메타와 연결되어 추적 가능하다.  ## 예외사항  - 설정 파일 누락·손상 - 설정 버전과 실제 구현 버전 불일치 - 런타임 중 설정 변경이 일부만 반영됨 - 설정 변경 이력이 남지 않음 - 테스트용 설정과 운영용 설정이 혼용됨 - 설정된 Adapter와 실제 환경 불일치  ## 아키텍처 관점 점검 포인트  ### 책임 귀속  - 설정 로드 책임 컴포넌트(Adapter 주입, 프로파일 로드 등)가 선언되어 있는가 - 기동 시 반영되는 설정과 런타임 반영 가능한 설정의 구분 기준이 명시되어 있는가 - 설정 변경 이력의 세션 메타 연결 책임이 귀속되어 있는가  ### 실패 모드·반응 경로  - 설정 파일 누락·손상 시의 기동 중단 또는 fallback 결정권이 명시되어 있는가 - 설정과 실제 환경 불일치의 검증·보고 경로가 정의되어 있는가 - 테스트용 프로파일과 운영용 프로파일의 혼용 방지 경로가 정의되어 있는가  ---  # Qualification (횡단)  ## 정상 흐름  - Adapter 교체·신규 구현 추가 시 Qualification 절차가 선행된다. - Mock/Replay 구현체도 동일한 포트 계약을 준수한다.  ## 예외사항  - 새 Adapter가 동일 파라미터에 대해 다른 결과를 반환한다. - Qualification을 통과하지 않은 구현체가 배포된다. - Mock과 실물의 동작 차이가 테스트에 노출되지 않는다. - 능력 선언이 실제 지원 범위와 맞지 않는다.  ## 아키텍처 관점 점검 포인트  ### 책임 귀속  - Adapter 교체 시 Qualification 수행 책임 주체가 선언되어 있는가 - Mock/Replay 구현체의 포트 준수 검증 기준이 명시되어 있는가 - 능력 선언의 실측 검증 책임이 귀속되어 있는가  ### 실패 모드·반응 경로  - Qualification 미통과 구현체의 운영 배포 차단 경로가 정의되어 있는가 - Mock과 실물의 동작 차이 발견 시의 보고 경로가 정의되어 있는가 - 능력 선언과 실측 결과 불일치 시의 감지 경로가 정의되어 있는가  ---  # 현재 아키텍처 구조가 커버해야 하는 핵심 질문  각 질문은 **책임 귀속**(ARCH에 선언되었는가)과 **실패 모드·반응 경로**(실패 시 어떻게 동작하는가) 두 층에서 모두 답해야 한다.  - Trigger Detector는 잘못된 시작/종료 신호와 복수 입력원을 감당할 수 있는가 - Capture Orchestrator는 단일/다중 카메라 상태를 집계하고 부분 장애·부분 종료를 표현할 수 있는가 - Camera Capture Worker는 프레임 취득·저장 경계 이전·장치 오류 보고를 분리된 책임으로 수행하며, 저장 경계를 로컬로 한정하는가 - Session Manager는 세션/이미지/메타데이터를 일관되게 묶고 정상/비정상 종료를 구분하며 불일치를 식별할 수 있는가 - Acquisition Parameter Manager는 파라미터 재조정 판단을 수집 흐름과 분리해 수행하고 프로파일의 저장·재적용을 지원할 수 있는가 - Camera Gateway Port는 SDK 교체·능력 차이·Mock/Replay를 흡수하며 Qualification을 전제로 운영되는가 - 시스템 전체의 관측 가능성(상태·오류·손실·성능)이 수집 흐름과 독립적으로 유지되는가 - 설정 관리가 기동·런타임·이력 축 모두에서 추적 가능한가 - 다중 카메라 시간 동기화 원칙 위반 상태를 탐지·기록·분리할 수 있는가  ---  # 한 줄 요약  현재 아키텍처가 충분한지 판단하려면, 정상 흐름만 보는 것이 아니라 **설치 실패, 트리거 오검출, 프레임 드랍, 저장 실패, 세션 불일치, 메타데이터 누락, 튜닝 실패, 다중 카메라 부분 장애, 관측 단절, 설정 불일치, Qualification 누락**까지 포함해 각 예외가 **어느 컴포넌트의 책임으로 귀속되는가**(책임 귀속)와 **어떤 반응 경로를 거치는가**(실패 모드·반응 경로) 두 층에서 모두 확인해야 한다. Worker가 카메라를 찾고 연결한다. - 카메라 파라미터를 초기화한다. - 스트리밍 준비가 완료된다. - Orchestrator가 전체 준비 상태를 파악한다.  ## 예외사항  - 카메라가 연결되지 않는다. - 동일 ID의 카메라가 둘 이상 잡히거나, 기대한 카메라가 다른 물리 장비로 연결된다. - 카메라 연결은 되지만 스트리밍 시작에 실패한다. - 특정 파라미터(exposure/gain/ROI/trigger mode)가 해당 SDK/장비에서 지원되지 않는다. - 파라미터 적용 중 일부만 반영되고 일부는 실패한다. - 초기화 시간 초과가 발생한다. - 카메라 연결은 됐지만 실제 프레임을 받지 못한다. - 다중 카메라일 때 일부만 ready 상태가 된다. - 시간 동기화가 필요한데 아직 공통 시간축 확보 전이다.  ## 아키텍처 관점 점검 포인트  ### 책임 귀속  - Worker 수준 오류와 Orchestrator 수준 오류의 경계가 선언되어 있는가 - 장치 능력(capabilities) 선언 경로가 포트 계약에 포함되어 있는가 - 일부 카메라 미준비 상태의 전체 상태 집계 책임이 Orchestrator에 귀속되어 있는가 - 초기화 타임아웃의 감지·보고 책임이 Worker에 귀속되어 있는가 - 카메라 논리 식별자와 물리 장비 매핑 책임 위치가 선언되어 있는가  ### 실패 모드·반응 경로  - 초기화 타임아웃 발생 시 Orchestrator의 반응 경로(세션 시작 차단, 다른 Worker 중단 등)가 명시되어 있는가 - 일부 Worker만 ready 상태일 때 수집 시작 여부 결정권이 귀속되어 있는가 - 파라미터 적용 부분 실패의 감지·보고 경로가 정의되어 있는가 - 동일 ID 중복 또는 잘못된 물리 매핑의 감지 경로가 정의되어 있는가  ---  # 검사 전 메타데이터 준비 단계  ## 정상 흐름  - 세션 시작 전 필요한 외부 정보와 사용자 입력 정보가 준비된다. - Session Manager가 세션에 연결할 정보를 확보한다.  ## 예외사항  - MES/상위 시스템 연결 실패로 작업지시 메타데이터를 가져오지 못한다. - PLC/상위 설비 신호는 오지만 제품/lot 정보는 없다. - 사용자 입력이 누락되거나 잘못 입력된다. - 외부 시스템 정보와 사용자 입력 정보가 서로 충돌한다. - 메타데이터가 늦게 들어와 촬영 시작 시점에 준비되지 않는다. - 외부 시스템 메타데이터가 변경되었는데 이미 열린 세션과 불일치한다. - source of truth가 불명확하여 어느 값을 기록해야 할지 모른다.  ## 아키텍처 관점 점검 포인트  ### 책임 귀속  - 메타데이터 4분류(External·User Input·Runtime Context·Derived)의 관리 책임이 Session Manager에 귀속되어 있는가 - 필수/선택 메타데이터 구분 책임이 Session Manager에 귀속되어 있는가 - 메타데이터 출처 간 충돌 중재 책임이 Session Manager에 귀속되어 있는가  ### 실패 모드·반응 경로  - 필수 메타 미준비 상태에서 세션 시작 차단·허용 결정권이 명시되어 있는가 - 출처 간 충돌 발생 시 적용되는 우선순위 규칙이 규약 수준에서 정의되어 있는가 - 세션 개시 후 외부 메타가 갱신된 경우의 처리 경로(무시·반영·기록)가 정의되어 있는가 - 외부 시스템 연결 실패 시 선택 메타로 강등 또는 시작 차단 결정 경로가 정의되어 있는가  ---  # 검사 시작 감지 단계  ## 정상 흐름  - Trigger Detector가 시작 신호를 감지한다. - 현재 시스템 상태를 반영해 유효한 시작 이벤트인지 판단한다. - Orchestrator에 시작 트리거를 전달한다.  ## 예외사항  - 시작 신호가 들어오지 않는다. - 시작 신호가 중복 발생한다. - 시작 신호와 종료 신호가 비정상 순서로 들어온다. - OCR이 잘못 읽어 false start가 발생한다. - Start mark가 오검출된다. - Start mark가 있어야 하는데 조명/속도 문제로 검출되지 않는다. - 수동 시작과 자동 시작이 동시에 들어온다. - 한 세션이 아직 닫히지 않았는데 다시 시작 신호가 들어온다. - 다중 트리거 소스가 서로 상충한다. - 시간 동기화 미확보 상태에서 시작 신호가 들어온다.  ## 아키텍처 관점 점검 포인트  ### 책임 귀속  - Trigger Detector의 중복 제거·상태 기반 필터링 책임이 선언되어 있는가 - 복수 트리거 입력의 중재 책임이 Trigger Detector 내부에 귀속되어 있는가 - 시간 동기화 미확보 시 세션 시작 차단 결정권이 특정 컴포넌트(Orchestrator 등)에 귀속되어 있는가 - 세션 미종료 상태에서의 재시작 차단 결정권이 귀속되어 있는가  ### 실패 모드·반응 경로  - 시작·종료 비정상 순서 신호의 무효화 경로가 정의되어 있는가 - 수동 시작과 자동 시작 동시 입력 시의 우선순위 규칙이 선언되어 있는가 - false start(OCR 오독, Start mark 오검출)를 이후 단계에서 정정하는 경로가 정의되어 있는가 - 시작 신호 장기 부재 시 상위 보고 경로가 정의되어 있는가  ---  # 프레임 수집 단계  ## 정상 흐름  - Worker가 카메라로부터 프레임을 취득한다. - Orchestrator가 전체 수집 상태를 모니터링한다. - Session Manager가 세션 스코프를 유지한다.  ## 예외사항  - 프레임 드랍이 발생한다. - SDK 콜백은 오지만 payload가 비정상이다. - 프레임 상태가 incomplete다. - 특정 시점부터 프레임이 멈춘다. - 카메라 연결은 살아 있지만 실제 데이터가 안 들어온다. - FPS가 기대치보다 낮다. - 광량 부족으로 노출이 부족하다. - blur로 불량이 판독 불가하다. - 카메라 내부 버퍼가 꽉 찬다. - Worker 내부 큐가 넘친다. - 다중 카메라에서 일부 카메라만 계속 프레임 누락이 난다. - 동기화된 프레임이라 기대했는데 실제 시간축이 어긋난다. - 장치 재연결이 필요하지만 세션 도중 발생한다.  ## 아키텍처 관점 점검 포인트  ### 책임 귀속  - Worker의 프레임 취득 실패 감지·보고 책임이 선언되어 있는가 - Orchestrator의 degraded 상태 인지 경로가 명시되어 있는가 - 프레임 유실·지연의 관측 기록 위치(세션 메타·시스템 로그)가 선언되어 있는가 - Acquisition Parameter Manager의 런타임 파라미터 재조정 지시 경로가 수집 흐름과 분리되어 있는가  ### 실패 모드·반응 경로  - SDK 콜백 payload 비정상 또는 incomplete 프레임의 Adapter 경계 도메인 에러 번역 경로가 정의되어 있는가 - 카메라 재연결 필요 시의 대응 정책 결정권(중단 · 재연결 시도 · degraded 진행)이 귀속되어 있는가 - Worker 내부 큐 오버플로우 시의 손실 기록 경로가 정의되어 있는가 - 프레임 장기 부재(연결은 살아 있으나 데이터 없음) 감지·보고 경로가 정의되어 있는가 - 다중 카메라에서 특정 카메라 드랍률만 비정상적으로 높을 때 degraded 선언 경로가 정의되어 있는가  ---  # 프레임 저장 및 세션 스테이징 단계  ## 정상 흐름  - Worker가 취득 직후 프레임을 세션 저장 경계에 기록한다. - 최소 메타데이터가 함께 기록된다. - Session Manager는 세션 수준 일관성을 관리한다.  ## 예외사항  - 저장 디스크 용량 부족 - 디스크 쓰기 속도 부족 - 파일 생성 실패 - 특정 프레임 저장 실패 후 다음 프레임은 들어온다 - 저장 지연이 누적되어 수집에 역류한다 - 파일명/경로 충돌 - 세션 스테이지 디렉터리 생성 실패 - 프레임은 저장됐지만 메타 기록이 누락된다 - 메타는 기록됐지만 프레임 파일이 없다 - 프로세스 비정상 종료로 일부 프레임만 저장된다 - 세션 중간에 머신 재부팅 또는 앱 크래시 발생 - 다중 카메라에서 카메라별 기록 개수가 맞지 않는다  ## 아키텍처 관점 점검 포인트  ### 책임 귀속  - Worker의 저장 내구성 책임이 선언되어 있는가 - Worker의 쓰기 대상을 로컬 저장 경계로 한정하는 원칙이 아키텍처 수준에서 선언되어 있는가 - 저장 용량 부족의 감지·보고 책임이 귀속되어 있는가 - Session Manager의 프레임/메타 불일치 식별 책임이 선언되어 있는가 - 비정상 종료 후 회복 가능한 세션 경계 유지의 책임이 귀속되어 있는가  ### 실패 모드·반응 경로  - 파일 생성 실패·경로 충돌 시 Worker의 실패 분류·보고 경로가 정의되어 있는가 - 저장 병목이 누적되어 수집 경로에 역류하려 할 때의 완화 구조가 명시되어 있는가 - 저장 용량 부족 보고 수신 시 Orchestrator의 반응 경로(세션 시작 차단·진행 중 세션 강제 종료)가 정의되어 있는가 - 프레임 ↔ 메타 불일치 발견 시의 세션 상태 표현 방식이 정의되어 있는가 - 프로세스 비정상 종료 또는 머신 재부팅 후 재기동 시 미완료 세션의 복구 책임이 귀속되어 있는가 - 다중 카메라 환경에서 카메라별 기록 개수 불일치의 감지·보고 경로가 정의되어 있는가  ---  # 검사 종료 감지 단계  ## 정상 흐름  - Trigger Detector가 종료 신호를 감지한다. - Orchestrator가 수집 종료를 지시한다. - Worker가 종료를 완료한다.  ## 예외사항  - 종료 신호가 오지 않는다. - 종료 신호가 너무 빨리 온다. - 종료 신호가 중복된다. - 시작 없이 종료만 온다. - 자동 종료가 실패해 세션이 계속 열린다. - 일부 카메라는 종료됐는데 일부는 계속 수집 중이다. - 타임아웃에 의해 강제 종료가 필요하다. - 운영자가 수동 종료를 눌렀지만 카메라는 계속 동작한다.  ## 아키텍처 관점 점검 포인트  ### 책임 귀속  - 종료 개시 책임의 이중 귀속(Trigger Detector 정상 경로, Orchestrator 강제 경로)이 선언되어 있는가 - 정상 종료·강제 종료의 유형 구분이 세션 생명주기에 선언되어 있는가 - 부분 종료 상태의 감지·수습 책임이 귀속되어 있는가  ### 실패 모드·반응 경로  - 종료 신호 장기 부재(타임아웃) 시 강제 종료 전환 결정권이 명시되어 있는가 - 수동 종료 명령 이후 카메라 계속 동작 시의 강제 정지 경로가 정의되어 있는가 - 시작 없이 종료만 도달한 경우의 무효화 경로가 정의되어 있는가 - 너무 이른 종료 신호(최소 세션 길이 미달)의 정책적 처리 경로가 정의되어 있는가  ---  # 세션 종료 및 Finalize 단계  ## 정상 흐름  - Session Manager가 세션을 닫는다. - 세션 수준 결과와 메타데이터를 정리한다. - 학습용 인계 가능한 상태가 된다.  ## 예외사항  - 세션 종료는 됐지만 일부 프레임이 아직 flush되지 않았다. - 세션 메타 확정 중 실패한다. - Derived metadata 계산 중 실패한다. - 종료 후 세션 상태와 실제 파일 상태가 불일치한다. - 강제 종료 세션을 정상 세션처럼 표시해버린다. - 다중 카메라 세션에서 일부 카메라 결과만 finalize된다. - 종료 직후 재기동 시 미완료 세션 복구가 필요하다.  ## 아키텍처 관점 점검 포인트  ### 책임 귀속  - Session Manager의 정상·비정상 세션 구분 관리 책임이 선언되어 있는가 - Finalize 실패의 세션 상태 표현 책임이 선언되어 있는가 - 세션 close와 프레임 flush 완료 간 동기화 책임이 명확히 귀속되어 있는가  ### 실패 모드·반응 경로  - Derived metadata 계산 실패 시 세션 상태 표현 방식이 정의되어 있는가 - 재기동 시 미완료 세션 복구 절차의 개시 책임이 명시되어 있는가 - 다중 카메라 환경에서 일부 카메라만 finalize된 상태의 수습 경로가 정의되어 있는가 - 강제 종료 세션이 정상 세션으로 잘못 표시되지 않도록 하는 기록 경로가 정의되어 있는가  ---  # 사후 검증 및 학습용 데이터 인계 단계  ## 정상 흐름  - 수집된 세션을 사람이 검토할 수 있다. - 학습용 데이터셋 후보로 인계할 수 있다.  ## 예외사항  - 실제 불량 제품을 찍었는데 불량이 관측되지 않는다. - 불량은 관측됐지만 해상도/조명/blur 때문에 판독이 어렵다. - 불량이 너무 작아 프레임 사이로 유실된다. - 실제로는 정상 제품이었는데 불량 샘플로 잘못 분류됐다. - 불량 위치를 알고 있었는데 세션 결과와 매칭이 안 된다. - 동일 조건 반복 테스트에서 결과가 달라 재현성이 없다. - 광학 조건 변경 후 이전 세팅과 비교가 안 된다. - 학습용으로 충분한 메타데이터가 없어서 데이터 활용성이 떨어진다.  ## 아키텍처 관점 점검 포인트  ### 책임 귀속  - Session Manager의 세션/메타/이미지 묶음 제공 책임이 선언되어 있는가 - 운영/검수 시스템이 연결될 확장 지점이 명시되어 있는가 - 튜닝/레시피 관리 기능이 추가될 확장 지점이 명시되어 있는가 - 세션 산출물이 read-only로 인계된다는 원칙이 선언되어 있는가  ### 실패 모드·반응 경로  - 외부 소비자(추론·검수 시스템)의 비정상 접근이 세션 상태에 영향을 주지 않는 구조가 명시되어 있는가 - 광학 조건 변경 전후 세션 비교를 위한 설정 이력 추적 경로가 정의되어 있는가 - 메타데이터 부족으로 인한 활용성 저하를 사전에 감지할 수 있는 경로가 정의되어 있는가  ---  # 튜닝 및 재세팅 관점 예외  ## 정상 흐름  - 기준 불량 샘플로 반복 테스트를 수행한다. - 광학 조건과 시스템 내부 수집 조건을 조절한다. - 재현성과 최소 검출 조건을 만족하는 세팅을 찾는다.  ## 예외사항  - 광학 조건 변경 후 시스템 내부 파라미터를 갱신하지 않았다. - 노출/게인/FPS/ROI 조합이 제품 조건과 맞지 않는다. - 카메라 SDK가 원하는 파라미터를 지원하지 않는다. - 수동 튜닝은 가능하지만 저장/재적용이 불가능하다. - 세팅값 변경 이력이 남지 않는다. - 제품 종류가 바뀌어도 이전 세팅을 그대로 사용한다. - 테스트용 세팅과 운영용 세팅이 혼용된다.  ## 아키텍처 관점 점검 포인트  ### 책임 귀속  - 튜닝/레시피 관리 계층의 추가 확장 지점이 아키텍처에 명시되어 있는가 - Adapter의 설정 적용 책임과 세팅 판단 책임의 분리 원칙이 선언되어 있는가 - 세팅 변경 이력의 세션 메타 연결 경로가 선언되어 있는가 - 파라미터 프로파일의 저장·재적용 책임이 귀속되어 있는가  ### 실패 모드·반응 경로  - 특정 파라미터가 Adapter 능력 선언에 포함되지 않을 때 상위의 적용 시도 차단 경로가 정의되어 있는가 - 테스트용 프로파일과 운영용 프로파일의 혼용 방지 경로(프로파일 구분 태그 등)가 정의되어 있는가 - 광학 조건만 변경되고 프로파일이 갱신되지 않은 상태의 감지·경고 경로가 정의되어 있는가  ---  # 다중 카메라 확장 시 추가 예외  ## 정상 흐름  - 여러 Worker가 동시에 동작한다. - Orchestrator가 전체 상태를 집계한다. - Session Manager가 카메라별 산출물을 일관되게 관리한다.  ## 예외사항  - 일부 카메라만 준비 완료 - 일부 카메라만 시작/종료 성공 - 카메라 간 시간축 불일치 - 동일 세션 내 카메라별 프레임 수 심각한 차이 - 특정 카메라의 드랍률만 비정상적으로 높음 - 카메라별 광학 조건이 달라 시인성이 다름 - 한 카메라 장애가 전체 세션을 중단해야 하는지 불명확 - 일부 카메라 degraded 상태에서 세션 계속 수행 여부가 모호  ## 아키텍처 관점 점검 포인트  ### 책임 귀속  - Orchestrator의 전역 상태 표현 책임이 선언되어 있는가 - 부분 장애 사실의 세션 메타 기록 책임이 Session Manager에 귀속되어 있는가 - 시간 동기화 실패의 신뢰도 상태 표현 책임이 귀속되어 있는가  ### 실패 모드·반응 경로  - 한 카메라 장애 시 전체 세션 중단 또는 degraded 진행 결정권이 귀속되어 있는가 - 카메라 간 프레임 수 심각한 차이의 감지·보고 경로가 정의되어 있는가 - 공통 시간축 이탈 구간의 프레임을 정상 데이터와 분리해 표시하는 기록 경로가 정의되어 있는가  ---  # 관측 가능성 (횡단)  ## 정상 흐름  - 시스템 상태와 세션 진행을 외부 관찰자가 확인할 수 있다. - 오류·손실·경고 이벤트가 사후 분석 가능한 형태로 기록된다. - 성능 지표(FPS·드랍률·지연·저장 속도)가 운영 중 확인 가능하다.  ## 예외사항  - 로그 저장 실패가 수집 흐름에 영향을 준다. - 세션 실패 원인을 사후에 추적할 수 없다. - 다중 카메라 중 특정 카메라만 degraded일 때 구분이 안 된다. - 메트릭 수집기가 부재할 때 시스템이 동작하지 않는다. - 오류 분류가 불명확해 설정·환경·장비 중 어느 문제인지 판별이 어렵다.  ## 아키텍처 관점 점검 포인트  ### 책임 귀속  - 프레임 유실·지연의 기록 위치(세션 메타 vs 시스템 로그)가 선언되어 있는가 - 컴포넌트별 상태(Worker 준비/수집/종료, Orchestrator 집계 상태, Session Manager 세션 상태)의 공표 경로가 명시되어 있는가 - 오류 유형 분류 체계(설정·환경·장비·세션·운영)의 보고 경로가 선언되어 있는가 - 관측 데이터의 세션 메타 결합 방식이 귀속되어 있는가  ### 실패 모드·반응 경로  - 로그·메트릭 쓰기 실패가 수집 흐름으로 역류하지 않는 구조가 명시되어 있는가 - 메트릭 수집 컴포넌트 부재 시 시스템 기능이 계속 동작하는 구조가 선언되어 있는가 - 관측 데이터 누락 시의 감지 경로가 정의되어 있는가  ---  # 설정 관리 (횡단)  ## 정상 흐름  - 설정 파일로부터 Adapter 선택·파라미터 프로파일·임계값이 로드된다. - 설정 변경이 세션 메타와 연결되어 추적 가능하다.  ## 예외사항  - 설정 파일 누락·손상 - 설정 버전과 실제 구현 버전 불일치 - 런타임 중 설정 변경이 일부만 반영됨 - 설정 변경 이력이 남지 않음 - 테스트용 설정과 운영용 설정이 혼용됨 - 설정된 Adapter와 실제 환경 불일치  ## 아키텍처 관점 점검 포인트  ### 책임 귀속  - 설정 로드 책임 컴포넌트(Adapter 주입, 프로파일 로드 등)가 선언되어 있는가 - 기동 시 반영되는 설정과 런타임 반영 가능한 설정의 구분 기준이 명시되어 있는가 - 설정 변경 이력의 세션 메타 연결 책임이 귀속되어 있는가  ### 실패 모드·반응 경로  - 설정 파일 누락·손상 시의 기동 중단 또는 fallback 결정권이 명시되어 있는가 - 설정과 실제 환경 불일치의 검증·보고 경로가 정의되어 있는가 - 테스트용 프로파일과 운영용 프로파일의 혼용 방지 경로가 정의되어 있는가  ---  # Qualification (횡단)  ## 정상 흐름  - Adapter 교체·신규 구현 추가 시 Qualification 절차가 선행된다. - Mock/Replay 구현체도 동일한 포트 계약을 준수한다.  ## 예외사항  - 새 Adapter가 동일 파라미터에 대해 다른 결과를 반환한다. - Qualification을 통과하지 않은 구현체가 배포된다. - Mock과 실물의 동작 차이가 테스트에 노출되지 않는다. - 능력 선언이 실제 지원 범위와 맞지 않는다.  ## 아키텍처 관점 점검 포인트  ### 책임 귀속  - Adapter 교체 시 Qualification 수행 책임 주체가 선언되어 있는가 - Mock/Replay 구현체의 포트 준수 검증 기준이 명시되어 있는가 - 능력 선언의 실측 검증 책임이 귀속되어 있는가  ### 실패 모드·반응 경로  - Qualification 미통과 구현체의 운영 배포 차단 경로가 정의되어 있는가 - Mock과 실물의 동작 차이 발견 시의 보고 경로가 정의되어 있는가 - 능력 선언과 실측 결과 불일치 시의 감지 경로가 정의되어 있는가  ---  # 현재 아키텍처 구조가 커버해야 하는 핵심 질문  각 질문은 **책임 귀속**(ARCH에 선언되었는가)과 **실패 모드·반응 경로**(실패 시 어떻게 동작하는가) 두 층에서 모두 답해야 한다.  - Trigger Detector는 잘못된 시작/종료 신호와 복수 입력원을 감당할 수 있는가 - Capture Orchestrator는 단일/다중 카메라 상태를 집계하고 부분 장애·부분 종료를 표현할 수 있는가 - Camera Capture Worker는 프레임 취득·저장 경계 이전·장치 오류 보고를 분리된 책임으로 수행하며, 저장 경계를 로컬로 한정하는가 - Session Manager는 세션/이미지/메타데이터를 일관되게 묶고 정상/비정상 종료를 구분하며 불일치를 식별할 수 있는가 - Acquisition Parameter Manager는 파라미터 재조정 판단을 수집 흐름과 분리해 수행하고 프로파일의 저장·재적용을 지원할 수 있는가 - Camera Gateway Port는 SDK 교체·능력 차이·Mock/Replay를 흡수하며 Qualification을 전제로 운영되는가 - 시스템 전체의 관측 가능성(상태·오류·손실·성능)이 수집 흐름과 독립적으로 유지되는가 - 설정 관리가 기동·런타임·이력 축 모두에서 추적 가능한가 - 다중 카메라 시간 동기화 원칙 위반 상태를 탐지·기록·분리할 수 있는가  ---  # 한 줄 요약  현재 아키텍처가 충분한지 판단하려면, 정상 흐름만 보는 것이 아니라 **설치 실패, 트리거 오검출, 프레임 드랍, 저장 실패, 세션 불일치, 메타데이터 누락, 튜닝 실패, 다중 카메라 부분 장애, 관측 단절, 설정 불일치, Qualification 누락**까지 포함해 각 예외가 **어느 컴포넌트의 책임으로 귀속되는가**(책임 귀속)와 **어떤 반응 경로를 거치는가**(실패 모드·반응 경로) 두 층에서 모두 확인해야 한다.
